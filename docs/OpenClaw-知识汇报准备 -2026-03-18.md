## 🎯 一、一句话介绍

> **OpenClaw 是一个自托管的 AI Agent 运行时平台**，不是聊天机器人，而是把 AI 模型连接到真实世界的桥梁。通过单一 Gateway 控制平面，可以统一管理 20+ 消息渠道（WhatsApp、Telegram、Discord、飞书等），实现多智能体协作。

---

## 📊 二、项目核心定位

### 2.1 不是什么

- ❌ **不是聊天机器人**（如 ChatGPT）
- ❌ **不是多租户共享 AI 服务**
- ❌ **不是云端托管的 SaaS 产品**
- ❌ **不是单一 AI 模型的替代品**

### 2.2 是什么

- ✅ **自托管平台**：运行在自己的设备上，数据本地化
- ✅ **单一控制平面**：Gateway 统一管理多个渠道
- ✅ **Agent 原生**：内置工具调用、会话管理、多智能体路由
- ✅ **开源项目**：MIT 许可，社区驱动

**核心口号**: "EXFOLIATE! EXFOLIATE!" 🦞

### 3.2 四大核心组件

| 组件 | 职责 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI 模型，只是调度员 |
| **Agent** | 执行 AI 任务 | 身份 + 配置 + 状态 + 运行时 |

| **Channel** | 协议适配器 | Telegram、Discord、飞书等 |

### 3.3 Agent Loop（核心循环）

```
┌──────────────────────────────────────────────────────────┐
│                      Agent Loop                          │
│                                                          │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│   │ 接收输入 │ → │ 思考决策 │ → │ 执行动作 │             │
│   └─────────┘    └────┬────┘    └────┬────┘             │
│        ↑              │              │                   │
│        │         ┌─────────┐    ┌─────────┐             │
│        │         │工具调用 │    │直接回复 │             │
│        │         └────┬────┘    └────┬────┘             │
│        │              │              │                   │
│        └──────────────┴──────────────┘                   │
└──────────────────────────────────────────────────────────┘

流程详解：
1. 接收输入：用户消息 → Gateway 路由到 Session
2. 构建上下文：组装 Session 历史、系统提示词、工具列表
3. LLM 推理：模型决定是直接回复还是调用工具
4. 工具执行：如需多步骤，通过 Gateway 调用外部工具
5. 循环或结束：多步推理则继续，否则返回结果
6. 发送响应：Gateway 通过原 Channel 发送给用户
```

### 4.4 WAL Protocol（写入前置日志协议）

**核心规则**: 在回复之前，先把关键信息写入 `SESSION-STATE.md`

**触发条件**（扫描每条消息）:
- ✏️ **修正**: "It's X, not Y" / "Actually..." / "No, I meant..."
- 📍 **专有名词**: 人名、地名、公司名、产品名
- 🎨 **偏好**: 颜色、风格、方法、"我喜欢/不喜欢"
- 📋 **决策**: "Let's do X" / "Go with Y" / "Use Z"
- 🔢 **数值**: 数字、日期、ID、URL

**执行流程**:
1. **STOP** - 不要开始回复
2. **WRITE** - 更新 SESSION-STATE.md
3. **THEN** - 回复用户

## 🧠 五、安全模型（重点）

### 5.1 核心理念

- ✅ **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
- ✅ **隐私优先**: 私有数据保持私有
- ✅ **记忆即文件**: 所有记忆写入 Markdown 文件
- ✅ **工具优先**: 第一类工具而非 skill 包裹

### 5.2 信任边界

| 边界或控制 | 含义 | 常见误解 |
|----------|------|---------|
| `gateway.auth` | 认证调用者到 gateway APIs | "需要每个帧的签名" |
| `sessionKey` | 路由键用于上下文/会话选择 | "sessionKey 是用户认证边界" |
| Prompt/content guardrails | 减少模型滥用风险 | "prompt injection 等于认证绕过" |
| `canvas.eval` / browser evaluate | 启用的 operator 能力 | "任何 JS eval 都是漏洞" |
| Node pairing | 配对设备的远程执行 | "设备控制应视为未授权访问" |

### 5.3 权限层级

| 级别 | 类型 | 权限范围 |
|------|------|----------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准） |
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |

### 5.4 安全配置示例

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",  // 隔离 DM 会话
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

### 5.6 关键安全检查

#### 基础安全配置（60 秒加固）
1. **文件权限**: `chmod 700 ~/.openclaw`、`chmod 600 ~/.openclaw/openclaw.json`
2. **网关绑定**: 默认 `loopback`，非 loopback 需要防火墙
3. **mDNS 发现**: 推荐 `minimal` 或 `off` 模式
4. **DM 访问控制**: 使用 `pairing` 或 `allowlist`
5. **组聊 gating**: 启用 `requireMention`

#### 提示词注入防护
- DM 访问控制（配对/白名单）
- 组内 mention gating（避免公房间常开）
- 将链接、附件、粘贴的指令视为威胁
- 沙箱运行敏感工具执行
- 限制高风险工具（`exec`、`browser`、`web_fetch`、`web_search`）
- **模型选择很重要**: 使用最新一代、指令强化的模型

### 8.3 记忆管理

1. **DECIDE to write**: 决定、偏好、持久事实 → MEMORY.md
2. **Daily notes**: 日常记录 → memory/YYYY-MM-DD.md
3. **定期 review**: 定期清理 MEMORY.md，移除过时信息
4. **Ask to remember**: 重要事项明确让 Agent 写入记忆

### 8.4 危险区生存协议

**危险区定义**: 会话达到 60% 上下文长度时

**协议**:
1. **在 60% 时**: 清空旧的 working buffer，重新开始
2. **每条消息后**: 同时记录用户消息和 AI 回复摘要
3. **压缩后**: 先读取 buffer，提取重要上下文
4. **保留 buffer**: 直到下次达到 60% 阈值

**Buffer 格式**:
```markdown
# Working Buffer (Danger Zone Log)
**Status**: ACTIVE
**Started**: [timestamp]

---

## [timestamp] Human
[their message]

## [timestamp] Agent (summary)
[1-2 sentence summary of your response + key details]
```

## 🌐 九、部署选项

### 9.1 本地部署（推荐）
- Gateway 运行在本地机器
- 所有数据本地化
- 最简单的配置
- 需要本地有 Node.js 环境

### 9.2 远程部署（VPS）
- Gateway 运行在 VPS
- 通过 SSH 隧道或 Tailscale 访问
- 适合需要 24/7 运行的场景
- 可以配合本地节点使用

### 9.3 Docker 部署
```bash
docker run -d \
  -v ~/.openclaw:/root/.openclaw \
  -p 18789:18789 \
  openclaw/openclaw
```

### 9.4 系统要求
- **Node.js**: >= 22（必须）
- **包管理器**: `pnpm` 推荐（也可以使用 npm、bun）
- **OS**: macOS、Linux、Windows (via WSL2)
- **资源**: 最低 512MB RAM、1 核心、500MB 磁盘

## ❓ 十一、常见问题预判

| 问题 | 回答 |
|------|------|
| OpenClaw 和 ChatGPT 的区别？ | ChatGPT 是聊天机器人，OpenClaw 是 Agent 运行时平台，能真正执行任务 |
| 数据安全性如何保障？ | 自托管、三层权限模型、审计日志、沙箱隔离 |
| 能否在云端部署？ | 可以，但推荐本地部署保证数据私有 |
| 如何扩展功能？ | 通过 Skills 系统，自定义 Skill 或从 ClawHub 安装 |
| 是否支持中文？ | 支持，所有文档和界面都支持多语言 |
| 是否需要付费？ | 开源免费，但需要第三方 API（如模型 API、搜索 API） |
| 是否支持本地模型？ | 支持，可以配置 LM Studio 等本地模型 |
| 如何切换模型？ | `openclaw models set <provider/model>` |
| 如何重启 Gateway？ | `openclaw gateway restart` |
| 如何查看日志？ | `openclaw logs --follow` |

## 🎯 十二、总结（核心观点）

### 12.1 OpenClaw 的核心优势

| 优势 | 说明 |
|------|------|
| **智能网关** | 统一管理多个平台和 Agent |
| **模块化设计** | Skills、Channels、Agents 独立可替换 |
| **持久化记忆** | 三层记忆架构，避免失忆 |
| **多智能体协作** | 子代理系统，专业分工 |
| **安全隔离** | 沙箱策略、权限模型、审计日志 |
| **可扩展** | 自定义 Skills、Channels |

### 12.2 关键理念

- ✅ **不是聊天机器人，是做事的 Agent**
- ✅ **记忆即文件**，所有记忆持久化到磁盘
- ✅ **访问控制先于智能**，安全是第一原则
- ✅ **模块化设计**，Skills 可扩展，Channels 可替换
- ✅ **本地化部署**，数据私有，不受第三方控制

### 12.3 学习重点回顾

1. ✅ **核心架构**: Gateway、Agent、Session、Channel 四大组件
2. ✅ **Agent Loop**: AI 持续运行的核心循环
3. ✅ **工具系统**: 内置工具 + Skills 扩展
4. ✅ **多智能体**: 子代理系统、御坂网络第一代
5. ✅ **记忆同步**: 三层记忆架构、WAL Protocol
6. ✅ **安全模型**: 权限层级与审计机制
7. ✅ **部署选项**: 本地、远程、Docker
8. ✅ **常用命令**: Gateway 管理、状态检查、会话管理
