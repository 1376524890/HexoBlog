## 🎯 一、OpenClaw 是什么

### 1.1 核心定位

OpenClaw 是一个**AI Agent 运行时平台**，类似于"个人版 Autogen"，但：

- ✅ **不是纯框架**，是**完整的系统**
- ✅ **不是纯聊天机器人**，是**可配置的 AI 助手**
- ✅ **不是纯自动化工具**，是**多智能体协作平台**

> **一句话总结**: OpenClaw = 智能体网关 + 多通道支持 + 技能系统 + 记忆系统 + 自动化能力

### 1.2 核心理念（必背）

1. **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
2. **隐私优先**：私有数据保持私有
3. **记忆即文件**：所有记忆写入磁盘 Markdown 文件
4. **工具优先**：第一类工具而非 skill 包裹

### 1.3 核心区别

| 对比项 | ChatGPT | OpenClaw |
|--------|---------|----------|
| **定位** | 聊天机器人 | Agent 运行时平台 |
| **能力** | 回答问题 | 真正执行任务 |
| **记忆** | 会话内记忆 | Markdown 文件持久化 |
| **工具** | 有限工具调用 | 丰富的工具系统 |
| **架构** | 单一模型 | 多智能体协作 |

## 🏗️ 二、核心架构

### 2.1 三层架构（必讲）

```
┌─────────────────────────────────────────┐
│      Agent Layer（智能层）               │
│  - Main Agent（主会话）                  │
│  - Subagents（子代理）                   │
│  - ACP Agents（编码代理）                │
└──────────────────┬──────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│   Gateway Layer（网关层）← 大脑！        │
│  - 控制平面、策略层、路由                │
│  - 身份认证、工具策略、会话管理          │
│  - 频道适配器 (Discord/WhatsApp 等)      │
│  ⚡ 不运行 AI 模型，只是调度员             │
└──────────────────┬──────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│     Node Layer（节点层）← 手脚           │
│  - 远程执行表面                          │
│  - 设备能力 (摄像头、屏幕、通知)         │
│  - macOS companion app                   │
└─────────────────────────────────────────┘
```

### 2.2 四大核心组件

| 组件 | 职责 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI 模型，只是调度员 |
| **Agent** | 执行 AI 任务 | 身份 + 配置 + 状态 + 运行时 |
| **Session** | 有状态容器 | 消息历史、上下文、工具状态 |
| **Channel** | 协议适配器 | 支持 10+ 平台 |

**重要**: Gateway 本身**不运行 AI 模型**，只是 AI 模型的"调度员"。

### 2.3 Agent Loop（智能体循环）

理解 OpenClaw 的关键是理解 **Agent Loop**：

```
┌──────────────────────────────────────────────┐
│                   Agent Loop                 │
│                                              │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐ │
│   │ 接收输入 │ → │ 思考决策 │ → │ 执行动作 │ │
│   └─────────┘    └────┬────┘    └────┬────┘ │
│        ↑              │              │       │
│        │              ↓              ↓       │
│        │         ┌─────────┐    ┌─────────┐ │
│        │         │工具调用 │    │直接回复 │ │
│        │         └────┬────┘    └────┬────┘ │
│        └──────────────┴──────────────┘       │
└──────────────────────────────────────────────┘
```

**流程**：
1. 接收输入：用户通过某个 Channel 发送消息
2. 思考决策：Gateway 组装 Context，LLM 决定动作
3. 执行动作：调用工具或直接回复
4. 循环或结束：如需多步推理则循环

### 3.3 子代理启动方式

**推荐方式（工具）**：
```python

  runtime: "subagent",
  agentId: "research-analyst",
  mode: "run",  # "run"=单次运行，"session"=持久会话
  task: "研究 XX 主题",
  label: "任务标签",
  thread: true  // 线程绑定
})
```

**通知机制**：子代理完成时会 **announce** 结果回主会话，包含状态、耗时、token 统计、estimated cost。

## 🧠 四、记忆系统

### 4.1 三层记忆架构

```
┌─────────────────────────────────────┐
│ Layer 1: 会话记忆 (Session Memory)    │
│ - 当前会话上下文                    │
│ - 临时决策和中间结果                │
└─────────────────────────────────────┘
         │
         ▼ 同步关键信息
┌─────────────────────────────────────┐
│ Layer 2: 任务记忆 (Task Memory)       │
│ - 任务计划文件                      │
│ - 子代理执行结果                    │
└─────────────────────────────────────┘
         │
         ▼ 同步重要发现
┌─────────────────────────────────────┐
│ Layer 3: 长期记忆 (Long-term Memory)  │
│ - MEMORY.md: 精选记忆               │
│ - memory/YYYY-MM-DD.md: 每日日志    │
└─────────────────────────────────────┘
```

### 4.2 记忆工具

- **`memory_search`**: 语义搜索，基于向量检索
- **`memory_get`**: 精准读取文件内容

### 4.3 记忆特性

1. ✅ **纯 Markdown 文件** - 易于编辑和版本控制
2. ✅ **向量检索** - 支持语义搜索
3. ✅ **混合搜索** - BM25 + 向量相似度
4. ✅ **自动压缩** - 临近 token 上限时自动整理记忆
5. ✅ **时间衰减** - 新内容优先，旧内容降权

### 4.4 记忆最佳实践

- **Write Immediately**: 及时写入，上下文最清晰时
- **WAL Before Responding**: 回复前先写入关键信息
- **Buffer in Danger Zone**: 60% 上下文时记录所有交互
- **Search Before Giving Up**: 尝试所有来源再放弃

## 🔐 六、安全机制

### 6.1 权限层级

| 级别 | 说明 |
|------|------|
| Level 5 | 主 Agent - 完全权限 |
| Level 4 | 可信子 Agent - 受限系统权限（需批准） |
| Level 3 | 标准子 Agent - 标准开发权限 |
| Level 2 | 受限子 Agent - 严格受限权限 |
| Level 1 | 只读子 Agent - 只读访问 |

### 6.2 安全命令

```bash
# 基本检查
openclaw security audit

# 深度检查
openclaw security audit --deep

# 自动修复
openclaw security audit --fix

# JSON 格式输出
openclaw security audit --json
```

### 6.3 安全最佳实践

1. ✅ **启用 DM 隔离** - 多用户场景必须启用 `dmScope: "per-channel-peer"`
2. ✅ **定期审计** - 使用 `openclaw security audit`
3. ✅ **限制通道** - 仅允许可信通道
4. ✅ **使用沙箱** - 敏感操作必须沙箱隔离
5. ✅ **最小权限** - 工具只开启必要的

### 6.4 核心安全原则

- Gateway 和 Node 属于同一信任域
- `sessionKey` 是路由键，不是认证 token
- Prompt guardrails 不是安全边界
- 本地 TUI `!` shell 是明确的操作者触发
