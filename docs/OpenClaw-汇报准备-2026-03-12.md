## 📝 详细内容要点

### 第一部分：OpenClaw 是什么？（5 分钟）

**核心定义（一句话）**：
> OpenClaw 是 **AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。

**与 ChatGPT 的区别**：

| 对比项 | ChatGPT | OpenClaw |
|--------|---------|----------|
| 定位 | 聊天机器人 | Agent 运行时平台 |
| 能力 | 生成文本 | 真正执行任务 |
| 记忆 | 会话内临时 | 持久化到磁盘文件 |
| 工具 | API 调用有限 | 文件系统、执行命令、浏览器控制等 |
| 部署 | 云端 SaaS | 本地部署，数据私有 |
| 安全性 | 受限于平台 | 多层次安全控制，审计完善 |

**四大核心理念（必背）**：
1. **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
2. **隐私优先**：私有数据保持私有
3. **记忆即文件**：所有记忆写入 Markdown 文件
4. **工具优先**：第一类工具而非 skill 包裹

**关键要点**：
- OpenClaw **不是聊天机器人**，而是能真正"做事"的 Agent 平台
- 把 AI 连接到真实世界的桥梁

### 第二部分：核心架构（10 分钟）

**三层架构**：

```
┌─────────────────────────────────────────────────────────┐
│              Agent Layer（智能层）                        │
│  - 主 Agent、子代理、编码代理                           │
└─────────────────────────┬───────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│           Gateway Layer（网关层）← 大脑！                  │
│  - 控制平面、路由、策略、会话管理                        │
│  - 频道适配器（Discord/WhatsApp/飞书等）                 │
│  ⚠️ Gateway 本身不运行 AI 模型，只是调度员                  │
└─────────────────────────┬───────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│              Node Layer（节点层）← 手脚                   │
│  - 远程执行表面                                          │
│  - 设备能力（摄像头、屏幕、通知、位置）                  │
└─────────────────────────────────────────────────────────┘
```

**四大核心组件**：

| 组件 | 作用 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI 模型，只是调度员 |
| **Agent** | 执行 AI 任务 | 身份 + 配置 + 状态 + 运行时 |
| **Session** | 有状态容器 | 消息历史、上下文、工具状态 |
| **Channel** | 协议适配器 | Telegram、Discord、飞书等 |

**Agent Loop（核心循环）**：
```
1. 接收输入 → 用户通过 Channel 发送消息
2. 构建上下文 → 组装 Session 历史、系统提示词、工具列表
3. LLM 推理 → 模型决定是"直接回复"还是"调用工具"
4. 工具执行 → 如需多步骤，通过 Gateway 调用外部工具
5. 循环或结束 → 多步推理继续，否则返回最终结果
6. 发送响应 → Gateway 通过原 Channel 发送给用户
```

**关键要点**：
- Gateway 是"大脑"，但本身不运行 AI 模型
- Agent 拥有决策权，主动决定需要什么信息、调用什么工具
- Session 管理上下文，通过 Compaction 机制解决 token 限制

### 第三部分：工具与技能系统（8 分钟）

**工具分类**：

| 工具类型 | 包含工具 | 作用 |
|----------|----------|------|
| **Runtime Tools** | exec, process, gateway | 执行命令、管理进程 |
| **Filesystem Tools** | read, write, edit, apply_patch | 文件操作 |
| **Session Tools** | sessions_list, sessions_history, sessions_spawn | 会话管理 |
| **Memory Tools** | memory_search, memory_get | 记忆操作 |
| **Web Tools** | web_search, web_fetch, tavily | 网络搜索 |
| **UI Tools** | browser, canvas | UI 自动化 |
| **Node Tools** | nodes | 设备控制 |
| **Messaging Tools** | message | 消息发送 |
| **Feishu Tools** | feishu_doc, feishu_drive, feishu_wiki 等 | Feishu 集成 |

**工具安全策略**：
- **工具 Profile**: `minimal` / `coding` / `messaging` / `full`
- **工具组 (Shorthands)**: `group:runtime`, `group:fs`, `group:sessions` 等
- **安全控制**: `tools.allow` / `tools.deny`, `sandbox`, `elevated`

**Skills 系统**：
- **Skill** 是专用任务的能力模块
- 提供特定领域的操作指导
- 模块化、可扩展、标准化
- 已安装 16 个 Skills：Hexo 博客、任务追踪、天气查询、多引擎搜索、飞书集成、主动代理、小红书运营等

**关键要点**：
- 工具基于 MCP（Model Context Protocol）标准
- Skills 是 MCP 的实现，每个 Skill 是一个独立包
- 工具安全通过 Profile 和规则控制

**子代理机制**：

- 启动方式：`sessions_spawn()` 或 `/subagents spawn`
- 完成时会 **announce** 结果回主会话

**调用示例**：
```python

  runtime: "subagent",
  agentId: "research-analyst",
  mode: "run",

})
```

**关键要点**：

### 第五部分：安全与最佳实践（5 分钟）

**安全模型**：

**权限层级**：
- Level 5: 主 Agent - 完全权限
- Level 4: 可信子 Agent - 受限系统权限（需批准）
- Level 3: 标准子 Agent - 标准开发权限
- Level 2: 受限子 Agent - 严格受限权限
- Level 1: 只读子 Agent - 只读访问

**安全原则**：
1. **Private things stay private**：私密信息不泄露
2. **Ask before acting externally**：外部行动前确认
3. **Never send half-baked replies**：不要发送半成品回复
4. **Be careful in group chats**：在群组中不要代表用户说话

**安全审计命令**：
```bash
openclaw security audit           # 基本检查
openclaw security audit --deep    # 深度检查
openclaw security audit --fix     # 自动修复
```

**最佳实践**：

1. **记忆管理**：
   - Write Immediately：及时写入
   - WAL Before Responding：回复前先写入关键信息
   - Buffer in Danger Zone：60% 上下文时记录所有交互

2. **工具使用**：
   - Profile 最小化：默认使用 minimal，按需开放
   - 沙箱优先：敏感操作使用沙箱环境
   - ask always：高风险操作设置 ask: always

3. **安全建议**：
   - 定期 audit：每月运行安全审计
   - 最小权限：按需开放工具
   - 强认证：使用长随机 token
   - 本地部署：Gateway 绑定到 loopback

**三层记忆架构**：
```
┌─────────────────────────────────────────┐
│ Layer 1: 会话记忆（Session Memory）       │
│ - 当前会话上下文                        │
└─────────────────────────────────────────┘
              ↓ 同步关键信息
┌─────────────────────────────────────────┐
│ Layer 2: 任务记忆（Task Memory）          │
│ - 任务计划文件                          │
└─────────────────────────────────────────┘
              ↓ 同步重要发现
┌─────────────────────────────────────────┐
│ Layer 3: 长期记忆（Long-term Memory）     │
│ - MEMORY.md：精选记忆                   │
│ - memory/YYYY-MM-DD.md：每日日志       │
└─────────────────────────────────────────┘
```

**关键要点**：
- 安全是第一原则
- 记忆即文件，所有记忆持久化到磁盘
- 所有操作有审计日志
