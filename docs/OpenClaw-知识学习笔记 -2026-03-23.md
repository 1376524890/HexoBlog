## OpenClaw 是什么？

**一句话介绍**: OpenClaw 是 AI Agent 运行时平台，不是聊天机器人，而是把 AI 连接到真实世界的桥梁。

### 核心定义

OpenClaw 是一个**自托管的网关**（self-hosted gateway），它将你最爱的聊天应用（WhatsApp、Telegram、Discord、iMessage 等）与 AI 编码代理（如 Pi）连接起来。你在自己的机器（或服务器）上运行一个单一的 Gateway 进程，它成为聊天应用和始终可用的 AI 助手之间的桥梁。

**目标用户**: 想要拥有一个可以从任何地方访问的个人 AI 助手的开发者和高级用户，同时不牺牲数据控制权或不依赖托管服务。

### 核心优势

1. **自托管**: 在你的硬件上运行，遵循你的规则
2. **多渠道**: 单个 Gateway 同时服务 WhatsApp、Telegram、Discord 等多个渠道
3. **Agent 原生**: 为编码代理构建，支持工具使用、会话、记忆和多智能体路由
4. **开源**: MIT 许可，社区驱动

### 需要的条件

- Node 24（推荐），或 Node 22 LTS（22.16+）以兼容
- 来自你选择的提供商的 API 密钥
- 5 分钟时间
- 最佳质量和安全：使用最新的生成式最强模型

## 核心理念

### 1. Access control before intelligence（访问控制先于智能）⭐⭐⭐⭐⭐

这是 OpenClaw 的核心原则。在 AI 能够访问任何数据或执行任何操作之前，必须先定义好权限边界。

### 2. 隐私优先

私有数据保持私有。你的 Gateway 运行在本地或受控服务器上，数据不会流向第三方。

### 3. 记忆即文件

所有记忆写入 Markdown 文件。这确保：
- 记忆持久化，会话重启后不会丢失
- 人类可读，可以随时查看和编辑
- 版本可控，可以通过 Git 管理

### 4. 工具优先

第一类工具而非 skill 包裹。工具是直接使用的能力，而不是通过 skill 系统间接调用。

## 四大核心组件

### 1. Gateway（网关）

**作用**: 控制平面、路由器、调度员

**职责**:
- 维护 provider 连接
- 暴露 typed WS API（请求、响应、服务端推送事件）
- 验证输入帧是否符合 JSON Schema
- 发出事件：`agent`、`chat`、`presence`、`health`、`heartbeat`、`cron`

**技术细节**:
- 单一长期运行的 Gateway 拥有所有消息表面
- 控制平面客户端（macOS 应用、CLI、Web UI、自动化）通过 WebSocket 连接到 Gateway
- 默认端口：127.0.0.1:18789
- 通过 SSH 或 Tailscale 支持远程访问

### 2. Agent（智能体）

**作用**: 执行 AI 任务

**职责**:
- 身份 + 配置 + 状态 + 运行时
- 接收输入 → 思考决策 → 执行动作 → 循环或发送响应（Agent Loop）
- 管理子代理和工具调用

**核心概念**:
- **Agent ID**: 唯一标识符
- **Workspace**: 工作空间，包含 AGENTS.md、SOUL.md 等配置文件
- **Session**: 有状态容器
- **Tools**: 可调用的工具集合

### 3. Session（会话）

**作用**: 有状态容器，用于维护对话上下文

**关键点**:
- 消息历史、上下文、工具状态
- 每个 Agent 可以有多个 Session
- Session 存储在磁盘上（JSONL 格式）
- 支持自动清理和压缩

**Session Key 格式**:
| 类型 | 格式 |
|------|------|
| 直接聊天 | `agent:<agentId>:main` 或 `agent:<agentId>:direct:<peerId>` |
| 群组聊天 | `agent:<agentId>:<channel>:group:<id>` |
| 频道聊天 | `agent:<agentId>:<channel>:channel:<id>` |
| Cron 任务 | `cron:<jobId>` |
| Webhook | `hook:<uuid>` |
| Node 运行 | `node-<nodeId>` |

**会话范围配置（dmScope）**:
- `main`（默认）：所有 DM 共享主会话
- `per-peer`：按发送者 ID 隔离
- `per-channel-peer`：按渠道 + 发送者隔离（推荐用于多用户信箱）
- `per-account-channel-peer`：按账户 + 渠道 + 发送者隔离（推荐用于多账户信箱）

**安全提示**: 如果你的 Agent 可以接收来自多个人的 DM，强烈建议启用安全 DM 模式（`dmScope: per-channel-peer`）。否则所有用户共享相同的对话上下文，可能导致隐私泄露。

### 4. Channel（渠道）

**作用**: 协议适配器，连接不同的聊天应用

**支持的渠道**:
- WhatsApp（通过 Baileys）
- Telegram（通过 grammY）
- Discord
- iMessage（通过 BlueBubbles）
- Slack
- Signal
- IRC
- Matrix
- Mattermost（通过插件）
- Feishu（飞书）
- Google Chat
- LINE
- Microsoft Teams
- 更多...

**Channel 职责**:
- 协议适配（不同聊天应用的 API 差异）
- 消息转换（将原始消息转换为 OpenClaw 格式）
- 事件订阅（监听新消息、用户在线状态等）

## 记忆系统

### 三层记忆架构

```
Layer 1: 会话记忆（Session Memory）
- 当前会话上下文
- 临时决策和中间结果
    ↓ 同步关键信息
Layer 2: 任务记忆（Task Memory）
- 任务计划文件
- 子代理执行结果
    ↓ 同步重要发现
Layer 3: 长期记忆（Long-term Memory）
- MEMORY.md：精选记忆
- memory/YYYY-MM-DD.md：每日日志
```

### 记忆文件说明

#### 1. MEMORY.md（精选记忆）
- **ONLY load in main session**（直接聊天与御坂大人）
- **DO NOT load in shared contexts**（Discord、群聊、其他会话）
- 这是**安全**考虑——包含不应该泄露给陌生人的个人上下文
- 可以**读取、编辑和更新**MEMORY.md
- 写入重要事件、想法、决策、意见、教训
- 是精选记忆——精华，不是原始日志
- 随着时间的推移，回顾我的日常文件并更新 MEMORY.md 值得保留的内容

#### 2. memory/YYYY-MM-DD.md（每日日志）
- 每天创建一个新的日志文件
- 记录当天的活动、决策、问题
- 是 raw logs（原始日志）
- 定期回顾并更新 MEMORY.md

#### 3. 会话记忆（Session Memory）
- 当前会话的上下文
- 临时决策和中间结果
- 存储在 JSONL 格式的转录文件中
- 支持自动清理和压缩

### 记忆维护（Heartbeat）

定期（每几天）使用 heartbeat 来：
1. 阅读最近的 `memory/YYYY-MM-DD.md` 文件
2. 识别值得长期保存的重要事件、教训或见解
3. 更新 `MEMORY.md` 中的精华内容
4. 删除 MEMORY.md 中不再相关的过时信息

## Cron 任务系统

### Cron 与 Heartbeat 的区别

**使用 Heartbeat 当**:
- 多个检查可以批量处理（邮件 + 日历 + 通知一次完成）
- 需要近期消息的对话上下文
- 时间可以稍有漂移（每~30 分钟即可）
- 想通过组合定期检查减少 API 调用

**使用 Cron 当**:
- 精确时间很重要（"每周一 9:00 整"）
- 任务需要与主会话历史隔离
- 想要为任务使用不同的模型或思考级别
- 一次性提醒（"20 分钟后提醒我"）
- 输出应直接发送到渠道而不涉及主会话

### Cron 配置示例

**一次性任务**:
```bash
openclaw cron add \
  --name "Reminder" \
  --at "2026-03-21T09:00:00Z" \
  --session main \
  --system-event "Reminder: submit report." \
  --wake now
```

**周期性任务**:
```bash
openclaw cron add \
  --name "Daily brief" \
  --cron "0 8 * * *" \
  --session isolated \
  --message "Summarize yesterday's work." \
  --announce \
  --channel feishu \
  --to "oc_xxx"
```

**使用环境变量**:
```bash
openclaw cron add \
  --name "Backup check" \
  --cron "0 0 * * *" \
  --session isolated \
  --message "Check backup status" \
  --light-context
```

### Cron 命令

```bash
openclaw cron add              # 添加任务
openclaw cron list             # 列出任务
openclaw cron run <id>         # 立即运行
openclaw cron edit <id>        # 编辑任务
openclaw cron remove <id>      # 删除任务
```

## Session 管理

### Session 生命周期

**重置策略**:
- **每日重置**: 默认在 Gateway 主机时间凌晨 4 点
- **空闲重置**: `idleMinutes` 增加一个滑动空闲窗口
- **触发重置**: `/new` 或 `/reset` 启动新会话

**重置配置**:
```json5
{
  session: {
    reset: {
      mode: "daily",           // daily | idle
      atHour: 4,               // 每天凌晨 4 点
      idleMinutes: 120,        // 空闲 2 小时
    },
    resetTriggers: ["/new", "/reset"],
  },
}
```

**重置类型覆盖**:
```json5
{
  session: {
    resetByType: {
      thread: { mode: "daily", atHour: 4 },
      direct: { mode: "idle", idleMinutes: 240 },
      group: { mode: "idle", idleMinutes: 120 },
    },
    resetByChannel: {
      discord: { mode: "idle", idleMinutes: 10080 },
    },
  },
}
```

### Session 维护

**维护配置**:
```json5
{
  session: {
    maintenance: {
      mode: "enforce",         // warn | enforce
      pruneAfter: "45d",       // 清理 45 天前的会话
      maxEntries: 800,         // 最多 800 个会话
      rotateBytes: "20mb",     // 超过 20MB 时轮换
      resetArchiveRetention: "14d",
    },
  },
}
```

**维护命令**:
```bash
# 预览清理效果
openclaw sessions cleanup --dry-run

# 执行清理
openclaw sessions cleanup --enforce
```

### Session 管理命令

```bash
openclaw sessions --json      # 列出会话
openclaw sessions cleanup     # 清理会话
```

## 常见问题

### Q1: OpenClaw 和 ChatGPT 的区别？

**A**: ChatGPT 是聊天机器人，只能对话；OpenClaw 是 Agent 运行时平台，能够真正执行任务、访问本地文件、运行代码、调用工具。

### Q2: 数据安全性如何保障？

**A**: 
- 自托管：运行在本地或受控服务器
