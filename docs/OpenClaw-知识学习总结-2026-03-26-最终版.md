## 🎯 核心知识点体系

### 一、OpenClaw 是什么？⭐⭐⭐⭐⭐

#### **一句话定义**
> **OpenClaw 是 AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。  
> 它不是聊天机器人，而是把 AI 连接到真实世界的桥梁。

#### **官方描述**
> OpenClaw 是一个**自托管网关（self-hosted gateway）**，将您最喜欢的聊天应用（WhatsApp、Telegram、Discord、iMessage 等）与 AI 编码智能体连接起来。您在自己机器上运行单一 Gateway 进程，它成为您的聊天应用和始终可用的 AI 助手之间的桥梁。

#### **四大核心理念（必背）** ⭐⭐⭐⭐⭐
1. **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
2. **隐私优先**：私有数据保持私有
3. **记忆即文件**：所有记忆写入 Markdown 文件
4. **工具优先**：第一类工具而非 skill 包裹

#### **目标用户**
- 希望拥有个人 AI 助力的开发者和高级用户
- 可以从任何地方向其发送消息
- 不希望在数据控制权上妥协或依赖托管服务

#### **核心竞争力**
- ✅ **自托管**：在您的硬件上运行，您的规则
- ✅ **多通道**：单个 Gateway 同时服务 WhatsApp、Telegram、Discord 等
- ✅ **Agent 原生**：为编码 Agent 构建，具有工具使用、会话、记忆和多 Agent 路由
- ✅ **开源**：MIT 许可，社区驱动

### 二、三层架构体系 ⭐⭐⭐⭐⭐

```
┌─────────────────────────────────────────────────────────┐
│              Agent Layer（智能层）                        │
│  - 主 Agent、Subagents、ACP Agents                        │
│  - 执行 AI 任务，拥有决策权                             │
│  - 每个 Agent 完全隔离：workspace、auth、sessions     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│           Gateway Layer（网关层）← 大脑！                  │
│  - 控制平面、策略层、路由                               │
│  - 身份认证、工具策略、会话管理                         │
│  - 频道适配器（15+ 个聊天平台）                          │
│  - WebSocket 协议统一控制平面                           │
│  ⚠️ Gateway 本身不运行 AI 模型，只是调度员                  │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│              Node Layer（节点层）← 手脚                   │
│  - 远程执行表面                                         │
│  - 设备能力（摄像头、屏幕、通知、位置）                 │
│  - macOS companion app, iOS/Android nodes               │
│  - 可远程执行 system.run                                │
└─────────────────────────────────────────────────────────┘
```

#### **记忆口诀**：
**智能层（脑）→ 网关层（路由）→ 节点层（手）**

#### **Gateway 架构深化（2026-03-23 新增）** ⭐⭐⭐⭐⭐

**单一长期运行**: 一个 Gateway 控制单一 Baileys 会话

**控制平面**: WebSocket 连接，typed API（JSON Schema 验证）

**事件类型**: `agent`、`chat`、`presence`、`health`、`heartbeat`、`cron`

**连接生命周期**:
```
Client -> Gateway: req:connect
Gateway -> Client: res (ok) + presence + health snapshot
Client -> Gateway: req:agent
Gateway -> Client: event:agent (streaming) + res:agent (final)
```

**安全机制**:
- 设备身份：所有连接包含 device identity
- 配对流程：新设备需要批准，颁发 device token
- 本地连接：loopback 可自动批准
- 签名验证：所有连接必须签名 connect.challenge

### 三、多智能体系统（Multi-Agent Routing）⭐⭐⭐⭐⭐

#### **什么是"一个 Agent"？**
一个**Agent**是完全隔离的大脑，拥有：
- **Workspace**（文件、AGENTS.md/SOUL.md/USER.md、本地笔记、persona 规则）
- **状态目录**（`agentDir`）用于认证配置、模型注册、每 Agent 配置
- **会话存储**（会话历史 + 路由状态）

**认证配置**是**每 Agent**的。每个 Agent 读取自己的：
```text
~/.openclaw/agents/<agentId>/agent/auth-profiles.json
```

**核心原则**: 主 Agent 凭据不会自动共享。永远不要在多个 Agent 间重用 `agentDir`（会导致认证/会话冲突）。

#### **路径映射速查**
| 配置项 | 默认路径 |
|--------|----------|
| Config | `~/.openclaw/openclaw.json` |
| State dir | `~/.openclaw` |
| Workspace | `~/.openclaw/workspace` |
| Agent dir | `~/.openclaw/agents/<agentId>/agent` |
| Sessions | `~/.openclaw/agents/<agentId>/sessions` |

#### **单 Agent 模式（默认）**
如果不做任何配置，OpenClaw 运行单个 Agent：
- `agentId` 默认为 **`main`**
- Sessions 键为 `agent:main:<mainKey>`
- Workspace 默认为 `~/.openclaw/workspace`
- State 默认为 `~/.openclaw/agents/main/agent`

#### **创建多个 Agent 工作区**
```bash
openclaw agents add coding
openclaw agents add social
```

#### **路由规则（消息如何选择 Agent）**
Bindings 是**确定性**的，**最具体优先**：
1. `peer` 匹配（精确 DM/组/频道 id）
2. `parentPeer` 匹配（线程继承）
3. `guildId + roles`（Discord 角色路由）
4. `guildId`（Discord）
5. `teamId`（Slack）
6. `accountId` 匹配（通道账户）
7. 通道级匹配（`accountId: "*"`)
8. 回退到默认 Agent（`agents.list[].default`，否则第一个列表项，默认 `main`）

**重要**: 如果多个 Bindings 在同一层级匹配，配置顺序中的第一个获胜。

#### **路由示例**
**单 WhatsApp 号码，多个人（DM 分流）**
```json5
{
  agents: {
    list: [
      { id: "alex", workspace: "~/.openclaw/workspace-alex" },
      { id: "mia", workspace: "~/.openclaw/workspace-mia" },
    ],
  },
  bindings: [
    {
      agentId: "alex",
      match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551230001" } },
    },
    {
      agentId: "mia",
      match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551230002" } },
    },
  ],
  channels: {
    whatsapp: {
      dmPolicy: "allowlist",
      allowFrom: ["+15551230001", "+15551230002"],
    },
  },
}
```

**同一通道，单个 Peer 到 Opus**
```json5
{
  agents: {
    list: [
      {
        id: "chat",
        name: "Everyday",
        workspace: "~/.openclaw/workspace-chat",
        model: "anthropic/claude-sonnet-4-6",
      },
      {
        id: "opus",
        name: "Deep Work",
        workspace: "~/.openclaw/workspace-opus",
        model: "anthropic/claude-opus-4-6",
      },
    ],
  },
  bindings: [
    {
      agentId: "opus",
      match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551234567" } },
    },
    { agentId: "chat", match: { channel: "whatsapp" } },
  ],
}
```
**Peer bindings 总是优先**，所以将它们放在通道级规则之上。

### 四、Session 管理系统 ⭐⭐⭐⭐⭐

#### **Session Key 格式**
| 场景 | Session Key 格式 |
|------|-----------------|
| 直接聊天 | `agent:<agentId>:main` 或 `agent:<agentId>:direct:<peerId>` |
| 群组聊天 | `agent:<agentId>:<channel>:group:<id>` |
| 频道聊天 | `agent:<agentId>:<channel>:channel:<id>` |
| Cron 任务 | `cron:<jobId>` |
| Webhook | `hook:<uuid>` |
| Node 运行 | `node-<nodeId>` |

#### **dmScope 配置（安全 DM 模式）**
```json5
{
  session: { dmScope: "per-channel-peer" },
}
```

**可选值**:
- `main`: 所有 DM 共享主会话（单用户场景）
- `per-peer`: 按发送者 ID 隔离
- `per-channel-peer`: 按渠道 + 发送者隔离（**多用户推荐**）⭐⭐⭐⭐⭐
- `per-account-channel-peer`: 按账户 + 渠道 + 发送者隔离（**多账户推荐**）⭐⭐⭐⭐⭐

**核心洞察**: 默认情况下，OpenClaw 将所有 DM 路由到主会话，以便您的助手跨设备和通道保持连续性。如果**多人**可以 DM 机器人（开放 DM 或多人允许列表），请考虑隔离 DM 会话以防止跨用户上下文泄露。

#### **Session 维护策略**
```json5
{
  session: {
    mode: "enforce",
    pruneAfter: "45d",
    maxEntries: 800,
    rotateBytes: "20mb",
    maxDiskBytes: "1gb",
    highWaterBytes: "800mb",
  },
}
```

**核心原则**: Session 是关键状态，所有会话状态存储在 sessions.json

### 六、Nodes 系统（节点）⭐⭐⭐⭐⭐

#### **什么是 Node？**
一个**节点**是配套设备（macOS/iOS/Android/headless），通过**WebSocket**连接到 Gateway（相同端口作为操作员），声明 `role: "node"`，并通过 `node.invoke` 暴露命令表面（如 `canvas.*`、`camera.*`、`device.*`、`notifications.*`、`system.*`）。

#### **核心概念**
- **Nodes 是外设，不是网关**。它们不运行网关服务。
- **Telegram/WhatsApp 等消息**落在**Gateway**上，而不是节点上。
- macOS 也可以运行在**节点模式**：菜单栏应用连接到 Gateway 的 WS 服务器，暴露本地 canvas/camera 命令作为节点。

#### **节点配对**
```bash
openclaw devices list
openclaw devices approve <requestId>
openclaw nodes status
openclaw nodes describe --node <idOrNameOrIp>
```

#### **远程节点主机（system.run）**
当您的 Gateway 在一台机器上，而您希望命令在另一台机器上执行时，使用**节点主机**。

**What runs where**:
- **Gateway host**: 接收消息，运行模型，路由工具调用
- **Node host**: 在节点机器上执行 `system.run`/`system.which`
- **Approvals**: 在节点主机上强制执行 `~/.openclaw/exec-approvals.json`

#### **启动节点主机（前台）**
```bash
openclaw node run --host <gateway-host> --port 18789 --display-name "Build Node"
```

#### **节点能力**
| 能力 | 描述 |
|------|------|
| `canvas.snapshot` | 屏幕截图 |
| `canvas.present` | 展示网页 |
| `canvas.eval` | 执行 JS |
| `camera.snap` | 拍照 |
| `camera.clip` | 视频录制 |
| `screen.record` | 屏幕录制 |
| `location.get` | 获取位置 |
| `device.status` | 设备状态 |
| `notifications.list` | 通知列表 |
| `photos.latest` | 最新照片 |

#### **安全控制**
- **Node 配对**: 所有连接包含 device identity，新设备需要批准

- **命令级别控制**: `gateway.nodes.allowCommands` 和 `gateway.nodes.denyCommands`

### 七、安全模型（Security）⭐⭐⭐⭐⭐

#### **5 级权限体系**
| 级别 | 类型 | 权限范围 |
|------|------|----------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准）|
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |

#### **安全审计命令**
```bash
openclaw security audit       # 基本检查
openclaw security audit --deep # 深度检查
openclaw security audit --fix  # 自动修复
openclaw security audit --json # JSON 格式
```

#### **加固基线配置（60 秒）**
```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer",
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

#### **核心安全原则**
1. **Identity first**: 先决定谁能与机器人交谈（DM 配对/允许列表/明确"开放"）
2. **Scope next**: 决定机器人可以在哪里行动（组允许列表 + 提及门控，工具，沙箱，设备权限）
3. **Model last**: 假设模型可以被操纵；设计使操纵具有有限的爆炸半径

#### **DM 访问模型**
所有当前 DM 通道支持 DM 策略（`dmPolicy` 或 `*.dm.policy`），在消息处理之前限制 DM：
- `pairing`（默认）：未知发送者收到配对码，机器人忽略其消息直到批准
- `allowlist`：未知发送者被阻止（无配对握手）
- `open`：允许任何人 DM（公开）。**需要**通道允许列表包含 `"*"`（明确选择）
- `disabled`：完全忽略入站 DM

#### **权限检查点（Security Audit）**
- **Inbound access**（DM 策略、组策略、允许列表）：陌生人能否触发机器人？
- **Tool blast radius**（高级工具 + 开放房间）：提示注入能否变成 shell/文件/网络行动？
- **Exec approval drift**（`security=full`、`autoAllowSkills`、解释器允许列表无 `strictInlineEval`）：主机执行护栏是否仍然起作用？
- **Network exposure**（Gateway 绑定/认证、Tailscale Serve/Funnel、弱/短认证令牌）
- **Browser control exposure**（远程节点、中继端口、远程 CDP 端点）
- **Local disk hygiene**（权限、符号链接、配置包含、"同步文件夹"路径）
- **Plugins**（扩展存在而无明确允许列表）

### 八、Cron vs Heartbeat ⭐⭐⭐⭐

#### **使用 Heartbeat 当**
- 多个检查可以批量处理（邮件 + 日历 + 通知）
- 需要对话上下文
- 时间可以稍有漂移（每~30 分钟）
- 想通过组合周期性检查减少 API 调用

#### **使用 Cron 当**
- 精确时间要求（"每周一 9:00 整"）
- 任务需要与主会话历史隔离
- 想要不同的模型或思考级别
- 一次性提醒（"20 分钟后提醒我"）
- 输出应直接传递到 channel 而不涉及主会话

#### **Heartbeat 状态追踪**
```json
{
  "lastChecks": {
    "email": 1703275200,
    "calendar": 1703260800,
    "weather": null
  }
}
```

### 九、记忆系统（Memory）⭐⭐⭐⭐⭐

#### **三层架构**
```
Layer 1: 会话记忆（Session Memory）
- 当前会话上下文
- 临时决策和中间结果
    ↓ 同步关键信息
Layer 2: 任务记忆（Task Memory）
- 任务计划文件
- 子代理执行结果
    ↓ 同步重要发现
Layer 3: 长期记忆（Long-term Memory）
- MEMORY.md：精选记忆
- memory/YYYY-MM-DD.md：每日日志
```

#### **WAL Protocol（写后读协议）** ⭐⭐⭐⭐⭐
- **STOP** — 不要立即回复
- **WRITE** — 更新记忆文件
- **THEN** — 回复用户

**触发条件**: 修正、专有名词、偏好、决策、编辑、数值

## 🎯 核心洞见总结（汇报重点）⭐⭐⭐⭐⭐

2. ✅ **记忆即文件**，所有记忆持久化到磁盘，不丢失
3. ✅ **安全第一**，多层权限控制和审计日志

9. ✅ **模型中立**，支持本地模型（vllm）和远程 API

### 深化核心洞见（2026-03-26 新增）⭐⭐⭐⭐⭐
1. ✅ **Gateway 不是 AI 模型**，只是调度员和控制平面
2. ✅ **Session 是关键状态**，所有会话状态存储在 sessions.json
3. ✅ **安全 DM 模式必要**：多用户场景必须启用 `dmScope: per-channel-peer`
4. ✅ **Session 维护重要**：定期清理防止磁盘膨胀
5. ✅ **工具优先设计**：工具是第一类能力，不是 skill 包裹
6. ✅ **Cron 与 Heartbeat 互补**: Cron 精确定时，Heartbeat 批量处理
7. ✅ **Nodes 是外设**：Gateway 是控制平面，Nodes 是远程执行表面
8. ✅ **设备配对必要**：所有连接包含 device identity，新设备需要批准
9. ✅ **WebSocket 协议统一**：控制平面 + 节点传输统一在 WS 协议上
10. ✅ **安全 DM 模式推荐**：`per-channel-peer` 是多用户最佳实践

## 🔑 关键命令行速查

### Gateway 管理
```bash
openclaw gateway status     # 检查状态
openclaw gateway start      # 启动
openclaw gateway stop       # 停止
openclaw gateway restart    # 重启
openclaw dashboard          # 打开控制 UI
```

### Agent 管理
```bash
openclaw agents add <name>  # 创建 Agent
openclaw agents list        # 列出所有 Agent

```

openclaw nodes status       # 节点状态
