### 2️⃣ Session 管理深化（官方文档）⭐⭐⭐⭐⭐

#### Session Key 格式（完整）

| 类型 | 格式 | 说明 |
|------|------|------|
| 直接聊天 (main) | `agent:<agentId>:main` | 所有 DM 共享 |
| 直接聊天 (per-peer) | `agent:<agentId>:direct:<peerId>` | 按发送者隔离 |
| 直接聊天 (per-channel-peer) | `agent:<agentId>:<channel>:direct:<peerId>` | **推荐：按渠道 + 发送者隔离** |
| 直接聊天 (per-account-channel-peer) | `agent:<agentId>:<channel>:<accountId>:direct:<peerId>` | 多账户推荐 |
| 群组聊天 | `agent:<agentId>:<channel>:group:<id>` | 群组隔离 |
| 频道聊天 | `agent:<agentId>:<channel>:channel:<id>` | 频道隔离 |
| Cron 任务 | `cron:<job.id>` | 独立任务 |
| Webhook | `hook:<uuid>` | Webhook 触发 |
| Node 运行 | `node-<nodeId>` | 节点命令 |

#### 安全 DM 模式（重要！）

```json5
{
  session: {
    dmScope: "per-channel-peer", // 推荐用于多用户场景
  },
}
```

**为什么重要**：
- **问题场景**: Alice 发送私密消息（医疗预约），Bob 问"我们刚才说什么？"
- **未设置 dmScope**: Bob 会看到 Alice 的上下文！
- **设置 dmScope**: Alice 和 Bob 的上下文完全隔离！

**dmScope 选项**:
- `main`（默认）: 所有 DM 共享主会话（单用户场景）
- `per-peer`: 按发送者 ID 隔离（跨渠道）
- `per-channel-peer`: 按渠道 + 发送者隔离（**多用户推荐**）⭐
- `per-account-channel-peer`: 按账户 + 渠道 + 发送者隔离（**多账户推荐**）⭐

#### Session 维护配置（生产环境）

```json5
{
  session: {
    maintenance: {
      mode: "enforce",        // 强制执行清理策略
      pruneAfter: "45d",      // 清理 45 天前的会话
      maxEntries: 800,        // 最多 800 个会话
      rotateBytes: "20mb",    // 超过 20MB 时轮换
      maxDiskBytes: "1gb",    // 磁盘上限
      highWaterBytes: "800mb", // 高水位线（80%）
      resetArchiveRetention: "14d",
    },
  },
}
```

**维护命令**:
```bash
# 预览清理
openclaw sessions cleanup --dry-run

# 执行清理
openclaw sessions cleanup --enforce
```

#### Session 生命周期（新增理解）

| 重置策略 | 触发条件 | 说明 |
|---------|---------|------|
| `daily` | 默认凌晨 4 点 | 按网关主机时区 |
| `idle` | 空闲 120 分钟 | 可配置 |
| `resetTriggers` | `/new` 或 `/reset` 命令 | 手动触发 |

**关键细节**:
- 重置是在**下次 inbound 消息**时评估，不是立即触发
- 如果同时配置 `daily` 和 `idle`，** whichever expires first **
- Cron 任务总是创建新的 `sessionId`（无 idle 复用）

### 3️⃣ Gateway 安全深化（官方文档）⭐⭐⭐⭐⭐

#### 安全模型前提（重要！）

**OpenClaw 的安全模型是**：
- ✅ **个人助手模式**：一个可信操作者边界，可能有多个 Agent
- ❌ **不是多租户安全边界**：不设计用于多个互不信任的用户共享一个 Gateway

**实际意义**：
- 如果一个 Gateway 可以接收多个人的消息，每个人都可以使用该 Agent 的权限集
- 如果需要敌对用户隔离，**必须运行多个 Gateway**（或至少多个 OS 用户/主机）
- `sessionKey` 是路由选择键，**不是用户认证边界**

#### 5 级权限体系（官方定义）

| 级别 | 类型 | 说明 |
|------|------|------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准）|
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |

#### 安全加固基线（60 秒）

```json5
{
  gateway: {
    mode: "local",
    bind: "loopback",
    auth: { mode: "token", token: "replace-with-long-random-token" },
  },
  session: {
    dmScope: "per-channel-peer", // 多用户场景必配！
  },
  tools: {
    profile: "messaging",
    deny: ["group:automation", "group:runtime", "group:fs", "sessions_spawn", "sessions_send"],
    fs: { workspaceOnly: true },
    exec: { security: "deny", ask: "always" },
    elevated: { enabled: false },
  },
  channels: {
    whatsapp: { dmPolicy: "pairing", groups: { "*": { requireMention: true } } },
  },
}
```

**这个配置的作用**：
1. 网关仅本地访问
2. DM 隔离（多用户安全）
3. 默认拒绝危险工具
4. 仅 Workspace 内文件读写
5. exec 需要询问
6. 禁用 elevated 工具
7. DM 需要配对

#### 安全审计命令（必会）

```bash
# 基本检查
openclaw security audit

# 深度检查
openclaw security audit --deep

# 自动修复
openclaw security audit --fix

# JSON 格式输出
openclaw security audit --json
```

**审计检查项**（重点）：
1. **Inbound access**（DM 政策、群组政策、允许列表）：陌生人能否触发 Bot？
2. **Tool blast radius**（elevated 工具 + 开放房间）：提示注入能否转化为 shell/文件/网络操作？
3. **Exec approval drift**（security=full, autoAllowSkills）：主机执行护栏是否有效？
4. **Network exposure**（Gateway bind/auth, Tailscale Serve/Funnel）：网络暴露情况
5. **Browser control exposure**（远程节点、relay 端口）：浏览器控制暴露
6. **Local disk hygiene**（权限、符号链接、配置包含）：磁盘安全

#### 重要安全概念（新增理解）

| 概念 | 含义 | 常见误解 |
|------|------|---------|
| `gateway.auth` | 认证调用者到 Gateway APIs | "每条消息都需要签名" |
| `sessionKey` | 路由键，用于上下文/会话选择 | "sessionKey 是用户认证边界" |
| Prompt/content guardrails | 减少模型滥用风险 | "提示注入 = 认证绕过" |
| `canvas.eval` / browser evaluate | 有意操作员功能 | "任何 JS eval 都是漏洞" |
| Node pairing and commands | 配对设备的操作员级远程执行 | "远程设备控制应视为未授权访问" |

### 4️⃣ 多智能体路由深化（官方文档）⭐⭐⭐⭐⭐

#### Agent vs Account 概念

| 概念 | 说明 | 路径示例 |
|------|------|---------|
| `agentId` | 一个"大脑"（workspace、per-agent auth、per-agent session store）| `~/.openclaw/agents/<agentId>/` |
| `accountId` | 一个渠道账号实例（如 WhatsApp account "personal" vs "biz"）| `channels.<channel>.accounts.<accountId>` |
| `binding` | 通过 `(channel, accountId, peer)` 路由到 `agentId` | 配置中的 `bindings` 数组 |

#### 路由规则优先级（最重要！）

**Most-specific wins** 原则：

1. `peer` match（精确 DM/群组/频道 id）⭐ 最高优先级
2. `parentPeer` match（线程继承）
3. `guildId + roles`（Discord 角色路由）
4. `guildId`（Discord 服务器）
5. `teamId`（Slack 团队）
6. `accountId` 匹配渠道账号
7. 渠道级匹配（`accountId: "*"`)
8. fallback 到默认 agent

**关键细节**：
- 如果一个 binding 设置了多个 match 字段（例如 `peer` + `guildId`），所有字段都必须匹配（AND 语义）
- 如果多个 bindings 在同一层级匹配，配置顺序中第一个胜出
- **Peer bindings 总是优先于 channel-wide 规则**，所以要把它们放在配置前面

#### 多 Agent 架构示例

**场景 1**: WhatsApp 日常聊天 + Telegram 深度工作

```json5
{
  agents: {
    list: [
      { id: "chat", name: "Everyday", model: "anthropic/claude-sonnet-4-6" },
      { id: "opus", name: "Deep Work", model: "anthropic/claude-opus-4-6" },
    ],
  },
  bindings: [
    { agentId: "chat", match: { channel: "whatsapp" } },
    { agentId: "opus", match: { channel: "telegram" } },
  ],
}
```

**场景 2**: 同一个渠道，一个 DM 路由到 Opus

```json5
{
  agents: {
    list: [
      { id: "chat", name: "Everyday" },
      { id: "opus", name: "Deep Work" },
    ],
  },
  bindings: [
    { 
      agentId: "opus",
      match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551234567" } }
    },
    { agentId: "chat", match: { channel: "whatsapp" } },
  ],
}
```

#### 安全沙箱配置（per-agent）

```json5
{
  agents: {
    list: [
      {
        id: "personal",
        workspace: "~/.openclaw/workspace-personal",
        sandbox: {
          mode: "off",  // 不沙箱化
        },
        tools: {},  // 所有工具可用
      },
      {
        id: "family",
        workspace: "~/.openclaw/workspace-family",
        sandbox: {
          mode: "all",     // 总是沙箱化
          scope: "agent",  // 每个 agent 一个容器
          docker: {
            setupCommand: "apt-get update && apt-get install -y git curl",
          },
        },
        tools: {
          allow: ["read"],                    // 仅允许 read
          deny: ["exec", "write", "edit", "apply_patch"],  // 拒绝危险工具
        },
      },
    ],
  },
}
```

8. ✅ **SessionKey 不是认证边界**: 只是路由选择键
