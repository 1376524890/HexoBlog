## 2. OpenMOSS 架构深度分析

### 2.1 核心定位

> **"AI manages AI"** - 多代理自主协作，人类只设定目标并检查结果。

OpenMOSS (Multi-agent Orchestration & Self-evolving System) 是专为 OpenClaw 构建的自组织多代理协作平台。

### 2.2 核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Human Admin                               │
│                  (设定目标，查看结果)                          │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    OpenMOSS API                              │
│                  (FastAPI :6565)                             │
│                                                              │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐            │
│  │Task     │ │Agent    │ │Review   │ │Score    │            │
│  │Scheduler│ │Manager  │ │System   │ │System   │            │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘            │
│                                                              │
│  Database: SQLite + SQLAlchemy (10 tables)                  │
└──────────────────────┬──────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┬──────────────┐
        ▼              ▼              ▼              ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ Planner │   │Executor │   │Reviewer │   │ Patrol  │
   │ Agent   │   │ Agent×N │   │ Agent   │   │ Agent   │
   └─────────┘   └─────────┘   └─────────┘   └─────────┘
```

### 2.3 四角色分工体系

| 角色 | 职责 | 描述 | Cron 唤醒 |
|------|------|------|----------|
| **Planner** | 创建任务、拆分模块、分配子任务、定义验收标准 | 项目负责人 - 全局规划与交付 | ✓ |
| **Executor** | 认领子任务、执行工作、提交交付物 | 执行者 - 产出代码和内容 | ✓ |
| **Reviewer** | 审核交付物质量、评分、批准或驳回重做 | 质量把关者 - 确保输出符合标准 | ✓ |
| **Patrol** | 监控系统健康、标记阻塞任务、发送警报 | 自动运维 - 防止任务卡死 | ✓ |

### 2.4 任务层次结构

OpenMOSS 使用三级任务结构管理复杂项目：

| 层级 | 描述 | 示例 |
|------|------|------|
| **Task** | 完整的项目目标 | 构建博客系统 |
| **Module** | 任务的功能分解 | 用户系统、文章管理、评论系统 |
| **Sub-Task** | 具体可执行的工作单元 | 实现用户注册 API、构建文章列表页 |

### 2.5 子任务状态机

```
pending → assigned → in_progress → review → done
              ↑                      │
              └──── rework ←─────────┘ (Rejected)
              │
              └── blocked → pending (Patrol 标记)
```

### 2.6 核心特性

#### 2.6.1 自组织协作
- Agent 通过 cron 定时唤醒
- 自主认领任务、执行、提交
- 无需人工编排

#### 2.6.2 闭环质量控制
```
执行 → 审核 → 评分 → [通过/重做]
```
- Reviewer 评分直接影响 Agent 排名
- 激励机制驱动输出质量

#### 2.6.3 自动巡逻与恢复
- Patrol Agent 持续监控系统
- 检测异常，标记阻塞任务
- 触发恢复机制
- Agent "死亡率" 降至 0%

#### 2.6.4 评分与激励系统
- Agent 有独立评分
- 排行榜竞争
- 审核结果直接影响排名

#### 2.6.5 可插拔技能
- OpenMOSS 只处理编排
- Agent 实际能力由其 Skill 决定
- 领域无关设计

### 2.7 Agent 工作流程

每个 Agent 通过 OpenClaw 的 cron 唤醒机制自主运行：

```
1. 唤醒 → 调用 OpenMOSS API 检查当前状态
2. 执行 → 执行角色特定动作
3. 回写 → 将结果写入 OpenMOSS
4. 休眠 → 等待下次唤醒
```

### 2.8 技术栈

| 层级 | 技术 | 描述 |
|------|------|------|
| 前端 | Vue 3 + shadcn-vue | 管理 WebUI |
| 后端 | FastAPI (:6565) | RESTful API |
| 数据库 | SQLite + SQLAlchemy | 10 张表 |
| Agent 运行时 | OpenClaw | 每个 Agent 是一个 OpenClaw 实例 |

## 3. AssemblyZero 架构深度分析

### 3.1 核心定位

> **企业级多代理编排平台** - 12+ Agent 并发，单一身份，完整治理，可衡量 ROI。

### 3.2 生产验证

**27 天处理 207 个 Issue (159 个已关闭)**：
- 平均速度：5.9 issues/天
- 峰值：55 issues/天

### 3.3 核心架构

```
┌─────────────────────────────────────────────────────────────┐
│                  Human Orchestrator                          │
│                 (人类意图与监督)                               │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                  LangGraph Workflows                         │
│              (5 个状态机，SQLite 检查点)                       │
└──────────────────────┬──────────────────────────────────────┘
                       │
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│Claude Agents│ │   Gemini    │ │ Governance  │
│  (12+ 并发) │ │ Verification│ │    Gates    │
└─────────────┘ └─────────────┘ └─────────────┘
```

### 3.4 多模型验证机制

**核心创新：Claude 构建，Gemini 审核**

| 阶段 | 模型 | 职责 |
|------|------|------|
| 实现 | Claude | 编写代码、设计文档 |
| 审核 | Gemini | 质量检查、安全审计 |

### 3.5 四大审核门禁

| 门禁 | 时机 | Gemini 检查内容 |
|------|------|-----------------|
| **Issue Review** | 工作开始前 | 需求清晰度、范围、风险 |
| **LLD Review** | 编码前 | 设计完整性、安全性、可测试性 |
| **Code Review** | PR 前 | 质量、模式、漏洞 |
| **Security Audit** | 合并前 | OWASP Top 10、依赖风险 |

### 3.6 Agent 角色体系

| 角色 | 上下文 | 任务 |
|------|--------|------|
| **Feature Agent** | 完整代码库 | 新功能、重构 |
| **Bug Fix Agent** | Issue 聚焦 | 特定 bug 调查 |
| **Documentation Agent** | 文档 + 代码 | README、Wiki、API 文档 |
| **Review Agent** | PR diff | 代码审核辅助 |
| **Audit Agent** | 合规性 | 安全、隐私审计 |

### 3.7 技术架构亮点

#### 3.7.1 单用户身份
- 所有 Agent 共享 API 凭证、Git 身份、权限模式
- 避免多身份混乱

#### 3.7.2 Worktree 隔离
- 每个 Agent 独立 git worktree
- 无冲突，干净的 PR

#### 3.7.3 凭证轮换
- API 配额耗尽时自动轮换
- 无缝切换

#### 3.7.4 会话协调
- Agent 可查看其他 Agent 工作状态
- 通过会话日志协调

### 3.8 LangGraph 状态机

| 工作流 | 节点数 | 目的 |
|--------|--------|------|
| **Issue** | 7 | Idea → 结构化 GitHub Issue |
| **Requirements** | 10 | Issue → 批准的 LLD (设计) |
| **Implementation Spec** | 7 | LLD → 具体实现指令 |
| **TDD Implementation** | 13 | Spec → 代码 + 测试 + PR |
| **Scout** | 可变 | 外部情报收集 |

### 3.9 RAG 系统

基于 ChromaDB 和 `all-MiniLM-L6-v2` 嵌入：
- **文档检索** (The Librarian) - 注入相关 ADR、标准、历史 LLD
- **代码库检索** (Hex) - AST 级 Python 索引 + 向量搜索
- **重复检测** (The Historian) - 检查相似历史 Issue
- **Token 预算管理** - 适配 LLM 提示限制
- **本地优先** - 数据不离开开发者机器

## 4. MAS-Zero 架构深度分析

### 4.1 核心定位

> **零监督多代理系统设计** - 无需验证集，元级自监督 MAS 设计。

### 4.2 核心创新

MAS-Zero 是一个 **元代理 (Meta-Agent)**，承担三个角色：
1. **设计 (Design)** - MAS 架构设计
2. **评估 (Evaluate)** - 性能评估
3. **验证 (Verify)** - 结果验证

### 4.3 算法流程

```
┌─────────────────────────────────────────────────────────────┐
│                     MAS-Zero                                 │
│                   (Meta-Agent)                               │
└──────────────────────┬──────────────────────────────────────┘
                       │
       ┌───────────────┴───────────────┐
       ▼                               ▼
┌─────────────────┐           ┌─────────────────┐
│  Meta-Iteration │           │ Self-Verification│
│                 │           │                 │
│ a) MAS-Design   │           │ 从所有候选方案中  │
│    任务分解     │───────────▶│ 选择最优结果     │
│    子MAS提议    │           │                 │
│                 │           │                 │
│ b) MAS-Feedback │           │                 │
│    可解性评估   │           │                 │
│    完整性评估   │           │                 │
└─────────────────┘           └─────────────────┘
```

### 4.4 核心步骤

#### 步骤 1: Meta-Iterations
1. **MAS-Design**:
   - 任务分解
   - 为每个子任务提议子 MAS
   - 将 MAS 设计框架化为代码生成

2. **MAS-Feedback**:
   - 评估生成 MAS 的可解性
   - 评估完整性
   - 使用 MAS 代码执行的中间输出进行评估

#### 步骤 2: Self-Verification
- 从元迭代过程中生成的所有候选方案中选择最合适的结果

### 4.5 关键特性

- **无需验证集** - 全程无需外部监督
- **元级自监督** - 在 MAS 设计上自监督
- **仅推理时** - 无需训练

### 4.6 性能表现

在数学推理、研究生级 QA、代码基准测试中，使用 GPT-4o、LLaMA3.3-70B、Qwen2.5-32B 取得强性能，无需任何外部监督。

## 6. 关键设计模式提取

### 6.1 角色分工模式

**核心思想**: 每个 Agent 有明确的角色和职责边界

```
Planner (规划) → Executor (执行) → Reviewer (审核) → Patrol (监控)
```

**御坂网络借鉴**:
- 御坂美琴一号 (Planner/Coordinator)
- 御坂妹妹 11-16 号 (Executor)
- 新增 Reviewer 角色
- 新增 Patrol 角色

### 6.2 闭环质量模式

**核心思想**: 执行后必须审核，审核不通过必须重做

```
提交 → 审核 → [通过/重做]
```

**御坂网络借鉴**:
- 添加 Reviewer Agent
- 建立评分机制
- 实现重做循环

### 6.3 多模型验证模式

**核心思想**: 不同模型有不同视角，相互验证提高质量

```
Model A (构建) → Model B (验证)
```

**御坂网络借鉴**:
- 可考虑引入 Gemini 作为验证层
- 或使用不同 Claude 模型版本进行交叉验证

### 6.4 状态机驱动模式

**核心思想**: 任务状态明确转换，状态机驱动工作流

```
pending → assigned → in_progress → review → done
```

**御坂网络借鉴**:
- 定义明确的任务状态
- 状态转换规则
- 阻塞与恢复机制

### 6.5 Cron 唤醒模式

**核心思想**: Agent 通过定时任务唤醒，检查状态，执行工作

```
Cron → 唤醒 → 检查状态 → 执行 → 回写 → 休眠
```

**御坂网络借鉴**:
- 已有 cron 机制
- 优化唤醒后的工作流程

### 6.6 RAG 知识增强模式

**核心思想**: 通过向量检索增强 Agent 上下文

```
任务 → 向量检索 → 注入相关文档 → 增强 Prompt
```

**御坂网络借鉴**:
- 建立 RAG 系统
- 历史任务检索
- 最佳实践注入

## 7. 技术债务识别

### 7.1 OpenMOSS 限制

1. **单一数据库** - SQLite 不适合大规模部署
2. **无模型降级检测** - 不检测模型版本降级
3. **简单评分系统** - 缺少贝叶斯动态更新

### 7.2 AssemblyZero 限制

1. **复杂度高** - 学习曲线陡峭
2. **企业绑定** - 为企业场景设计，个人使用重
3. **门禁延迟** - 多门禁增加周转时间

### 7.3 MAS-Zero 限制

1. **研究性质** - 主要是研究框架，生产可用性待验证
2. **计算密集** - 元迭代需要大量计算
3. **评估依赖** - 评估质量依赖执行中间输出

## 8. 总结与建议

### 8.1 核心发现

1. **角色分工是关键** - 明确的职责边界提高效率
2. **闭环质量是保障** - 审核机制确保输出质量
3. **状态机是骨架** - 明确的状态转换驱动工作流
4. **多模型是趋势** - 不同模型交叉验证提高可靠性

### 8.2 对御坂网络的建议

| P1 | 添加 Patrol 角色 | OpenMOSS |
| P1 | 建立 RAG 知识库 | AssemblyZero |
