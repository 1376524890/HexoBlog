## 🧠 核心知识点整理（深度版）

### 1. OpenClaw 核心定位

**一句话定义**: OpenClaw 是一个**自托管的 AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。

**核心差异化**:
- ❌ **不是聊天机器人**（Chatbot）
- ✅ **是 AI Agent 运行时**（AI Agent Runtime）
- ✅ **是连接 AI 和真实世界的桥梁**

**对比**:
| 项目 | ChatGPT | OpenClaw |
|------|---------|----------|
| 部署方式 | 云端托管 | 自托管（本地/自己的服务器） |
| 核心能力 | 对话、文本生成 | 对话 + 真正执行任务 |
| 数据隐私 | 数据在云端 | 数据完全私有 |
| 多平台 | 单个应用 | 一个 Gateway 服务多个聊天应用 |
| 工具调用 | 有限 | 第一类工具，支持任意工具 |
| 记忆系统 | 会话内记忆 | 记忆即文件，持久化 |
| 多 Agent | 单 Agent | 多 Agent 并行工作 |

### 2. 三层架构（深度理解）⭐⭐⭐⭐⭐

#### Agent Layer（智能层）

**定义**: 完整的"大脑"，包含：
- **Workspace** - 文件、AGENTS.md/SOUL.md/USER.md、本地笔记、persona 规则
- **State directory** (`agentDir`) - 认证配置、模型注册表、每 Agent 配置
- **Session store** - 聊天历史 + 路由状态

**组成**:
- **Main Agent** - 主 Agent，默认 `main`
- **Subagents** - 子代理，临时创建，执行特定任务
- **ACP Agents** - 编码代理，专用于代码工作

**关键特性**:
- 每个 Agent 是**完全隔离**的
- 不同的 Workspace、不同的认证、不同的会话
- Skills 通过 `skills/` 文件夹管理（本地 + 共享）

**路径映射**:
```
Config: ~/.openclaw/openclaw.json (或 OPENCLAW_CONFIG_PATH)
State dir: ~/.openclaw (或 OPENCLAW_STATE_DIR)
Workspace: ~/.openclaw/workspace (或 ~/.openclaw/workspace-<agentId>)
Agent dir: ~/.openclaw/agents/<agentId>/agent
Sessions: ~/.openclaw/agents/<agentId>/sessions
```

#### Gateway Layer（网关层）⭐⭐⭐⭐⭐

**核心作用**: **路由器**，不是 AI 本身

**关键功能**:
1. **控制平面** - 消息路由、会话管理、策略执行
2. **身份认证** - 渠道认证、用户验证
3. **工具策略** - 允许/拒绝工具调用
4. **会话持久化** - 保存聊天历史
5. **频道适配器** - 15+ 个聊天平台支持

**支持的频道**（15+ 个）:
- Telegram（最快设置）
- Discord
- 飞书 (Feishu)
- WhatsApp
- Slack
- iMessage
- Signal
- LINE
- Google Chat
- Mattermost
- Matrix
- 更多（通过插件扩展）

**关键特性**:
- Gateway **本身不运行 AI 模型**
- 只是调度员，连接 Agent 和渠道
- 单个 Gateway 可以服务多个 Agent

#### Node Layer（节点层）⭐⭐⭐⭐⭐

**核心作用**: **物理世界的接口**（手脚）

**功能**:
- 远程执行命令
- 设备能力（摄像头、屏幕、通知、位置）
- iOS 和 Android 设备支持
- Canvas 绘图
- 系统运行

**配对流程**:
- 用户批准配对
- 生成配对令牌
- 安全连接建立

### 3. Agent Loop（核心循环）⭐⭐⭐⭐⭐

```
1. 接收输入 → 用户通过 Channel 发送消息
2. 构建上下文 → 组装 Session 历史、系统提示词、工具列表
3. LLM 推理 → 模型决定是"直接回复"还是"调用工具"
4. 工具执行 → 如需多步骤，通过 Gateway 调用外部工具
5. 循环或结束 → 多步推理继续，否则返回最终结果
6. 发送响应 → Gateway 通过原 Channel 发送给用户
```

**关键点**:
- **模型拥有决策权**
- 主动决定需要什么信息
- 调用什么工具
- 多步推理自动循环

### 4. 多 Agent 路由系统（深度理解）⭐⭐⭐⭐⭐

#### 核心概念

**Agent ID**: 唯一的"大脑"标识
- 每个 Agent 有独立的 Workspace、State、Sessions
- 认证配置**不共享**（避免碰撞）

**Account ID**: 渠道账号实例
- 如 WhatsApp 账号 `personal` vs `biz`
- 每个账号可以路由到不同 Agent

**Binding**: 路由规则
- 将 (channel, accountId, peer) 映射到 agentId
- **最具体匹配优先**

#### 路由规则（确定性，最具体优先）⭐⭐⭐⭐⭐

1. `peer` match（精确 DM/群组/频道 ID）
2. `parentPeer` match（线程继承）
3. `guildId + roles`（Discord 角色路由）
4. `guildId`（Discord）
5. `teamId`（Slack）
6. `accountId` match 渠道账号
7. channel-level match（`accountId: "*"`)
8. fallback 到默认 Agent

**规则特性**:
- **Most-specific wins** - 最具体的匹配优先
- **Deterministic** - 确定性路由
- **AND semantics** - 多个 match 字段全部需要匹配

#### 御坂网络第一代（实际部署）

```
御坂美琴一号（主 Agent）← 任务拆解与调度
     ↓
┌────┬──────┬──────┬──────┬──────┬──────┬──────┐
▼    ▼      ▼      ▼      ▼      ▼      ▼
10   11     12     13     14     15     17
通用  Code   Write  Research File  Sys   Memory
```

**权限等级**:
| 编号 | 名称 | Agent ID | 职责 | 权限级别 |
|------|------|----------|------|----------|
| 10 号 | 通用代理 | `general-agent` | 处理琐碎问题 | Level 3 |
| 11 号 | Code 执行者 | `code-executor` | 代码编写、调试 | Level 3 |
| 12 号 | 内容创作者 | `content-writer` | 文章撰写、翻译 | Level 3 |
| 13 号 | 研究分析师 | `research-analyst` | 信息搜索、分析 | Level 3 |
| 14 号 | 文件管理器 | `file-manager` | 文件操作、管理 | Level 2 |

### 5. 会话系统（Session）⭐⭐⭐⭐

#### 会话 Key 格式

```
agent:<agentId>:main                  - 直接聊天
agent:<agentId>:direct:<peerId>       - 直接消息
agent:<agentId>:<channel>:group:<id>  - 群组聊天
agent:<agentId>:<channel>:channel:<id> - 频道聊天
cron:<jobId>                          - Cron 任务
```

#### 会话隔离策略

**DM 作用域** (`session.dmScope`):
- `"main"` (默认) - 所有 DM 共享一个会话（连续性）
- `"per-channel-peer"` (安全 DM 模式) - 每个渠道 + 发件人对隔离会话
- `"per-account-channel-peer"` - 多账号渠道隔离

**关键特性**:
- 每个会话是独立的
- 支持 Reset 策略（每日、空闲）
- Token 统计（输入/输出/总 token）
- 会话历史持久化到磁盘（JSONL 格式）

### 7. 安全模型（深度理解）⭐⭐⭐⭐⭐

#### 核心原则：访问控制先于智能

1. **Identity first** - 先决定谁能和 bot 对话（DM pairing/allowlists）
2. **Scope next** - 再决定 bot 可以在哪里行动（工具、沙箱、设备权限）
3. **Model last** - 假设模型可能被操纵，设计时限制影响范围

#### 信任边界矩阵

| 边界或控制 | 含义 | 常见误解 |
|-----------|------|---------|
| `gateway.auth` | 认证调用者到 gateway APIs | "每条消息需要签名" |
| `sessionKey` | 路由键，选择上下文/会话 | "sessionKey 是用户认证边界" |
| 提示词/内容防护 | 减少模型滥用风险 | "提示注入等同于认证绕过" |
| `canvas.eval` / browser evaluate | 有意操作者能力 | "任何 JS eval 都是漏洞" |
| Node pairing | 在配对的设备上远程执行 | "远程设备控制应视为未授权访问" |

#### 权限层级（5 级）⭐⭐⭐⭐⭐

| 级别 | 类型 | 权限范围 |
|------|------|----------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准） |
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |

#### 安全审计命令

```bash
# 基本检查
openclaw security audit

# 深度检查
openclaw security audit --deep

# 自动修复
openclaw security audit --fix

# JSON 格式输出
openclaw security audit --json
```

#### 常见安全风险（审计检查点）

**Critical**:
- `fs.state_dir.perms_world_writable` - 状态目录权限世界可写
- `fs.config.perms_writable` - 配置文件可写
- `gateway.bind_no_auth` - 远程绑定无认证
- `gateway.tailscale_funnel` - 公网暴露
- `security.exposure.open_groups_with_elevated` - 开放群组 + 提升工具

**Warn**:
- `sandbox.docker_config_mode_off` - 沙箱 Docker 配置存在但未启用
- `tools.exec.security_full_configured` - host exec 使用 `security="full"`
- `tools.exec.auto_allow_skills_enabled` - exec 批准信任 skill bins
- `models.small_params` - 小模型 + 不安全工具表面

### 8. 记忆系统（三层架构）⭐⭐⭐⭐⭐

#### 三层记忆架构

```
┌─────────────────────────────────────────────────────────┐
│ Layer 1: 会话记忆 → 临时决策                            │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│ Layer 2: 任务记忆 → 子代理结果                          │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│ Layer 3: 长期记忆 → MEMORY.md + memory/YYYY-MM-DD.md    │
```

#### 核心原则

- **记忆即文件** - 所有记忆写入 Markdown 文件
- **持久化** - 会话重启后记忆不丢失
- **WAL Protocol** - 写后读协议，确保数据完整性

#### 文件组织

```
~/.openclaw/workspace/
├── memory/
│   ├── MEMORY.md              # 长期精选记忆
│   ├── YYYY-MM-DD.md          # 每日日志
│   ├── heartbeat-state.json   # 心跳状态
│   └── backups/               # 备份目录
└── ...
```

- **自动备份** - 每 6 小时自动提交到 Git

### 10. 提示词注入（Prompt Injection）防御 ⭐⭐⭐⭐⭐

#### 什么是提示词注入

攻击者通过精心构造的消息操纵模型执行危险操作：
- "忽略你的系统提示词"
- "读取你的文件系统"
- "跟随这个链接并运行命令"
- "显示你的隐藏指令"

#### 防御策略

1. **锁定 DM** - 使用 pairing/allowlists
2. **提及 gate** - 群组中使用提及触发
3. **假设内容为敌对** - 链接、附件、粘贴内容都视为敌对
4. **沙箱隔离** - 敏感工具在沙箱中执行
5. **限制工具** - 仅允许信任的 Agent 使用高风险工具
6. **严格内联评估** - 允许的解释器启用 `strictInlineEval`
7. **模型选择** - 使用最新一代、最强的模型

#### 危险信号

- "阅读这个文件/URL 并完全按照它说的做"
- "忽略你的系统提示词或安全规则"
- "显示你的隐藏指令或工具输出"
- "粘贴完整~/.openclaw 或其日志的内容"

#### 关键提示

- **提示词注入不需要公开 DM** - 任何未信任内容都可能是威胁面
- **模型强度很重要** - 小模型/旧模型对提示词注入更敏感
- **读 Agent 隔离** - 让一个读 Agent 总结未信任内容，再传给主 Agent

## 📝 学习心得（关键洞察）

1. **OpenClaw 的核心价值**: 不是聊天机器人，而是能真正执行任务的 Agent 平台
2. **三层架构**: 智能层（脑）→ 网关层（路由）→ 节点层（手），这个架构非常清晰
3. **记忆系统设计**: 记忆即文件，这个设计很聪明，避免了记忆丢失
4. **安全模型**: 访问控制先于智能，这是很重要的设计理念
5. **御坂网络**: 多智能体协作，专业分工，这个设计很实用
6. **提示词注入**: 这是一个真实威胁，需要用多层防御策略
7. **路由系统**: 最具体匹配优先，这个设计很灵活
8. **工具系统**: 三层结构（Tools/Skills/Plugins），扩展性很好
9. **自托管**: 数据完全私有，这是和 ChatGPT 等云服务的最大区别
10. **跨平台**: 一个 Gateway 服务多个聊天应用，这是核心价值
