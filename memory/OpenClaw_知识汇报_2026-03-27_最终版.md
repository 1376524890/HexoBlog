## 📌 一句话定义（必背）⭐⭐⭐⭐⭐

> **OpenClaw 是 AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。

## 🔧 四大核心组件（必懂）⭐⭐⭐⭐⭐

| 组件 | 作用 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI 模型，只是调度员；默认端口 18789 |
| **Agent** | 执行 AI 任务 | Agent Loop：接收→思考→执行→循环；拥有决策权 |
| **Session** | 有状态容器 | 消息历史、上下文、工具状态；Session Key 格式规范 |
| **Channel** | 协议适配器 | 支持 15+ 个聊天平台：Telegram、Discord、飞书、WhatsApp 等 |

**Agent Loop（核心循环）⭐⭐⭐⭐⭐**:
1. 接收输入 → 用户通过 Channel 发送消息
2. 构建上下文 → 组装 Session 历史、系统提示词、工具列表
3. LLM 推理 → 模型决定是"直接回复"还是"调用工具"
4. 工具执行 → 如需多步骤，通过 Gateway 调用外部工具
5. 循环或结束 → 多步推理继续，否则返回最终结果
6. 发送响应 → Gateway 通过原 Channel 发送给用户

## 📊 Session 管理深化（2026-03-23 新增）⭐⭐⭐⭐⭐

### Session Key 格式

| 类型 | Session Key 格式 |
|------|----------------|
| 直接聊天 | `agent:<agentId>:main` 或 `agent:<agentId>:direct:<peerId>` |
| 群组聊天 | `agent:<agentId>:<channel>:group:<id>` |
| 频道聊天 | `agent:<agentId>:<channel>:channel:<id>` |
| Cron 任务 | `cron:<jobId>` |
| Webhook | `hook:<uuid>` |
| Node 运行 | `node-<nodeId>` |

### dmScope 配置（安全 DM 模式）⭐⭐⭐⭐⭐

| 配置 | 说明 | 适用场景 |
|------|------|----------|
| `main` | 所有 DM 共享主会话 | 单用户场景 |
| `per-peer` | 按发送者 ID 隔离 | 多用户场景（轻度）|
| `per-channel-peer` | 按渠道 + 发送者隔离 | **多用户推荐** ⭐⭐⭐⭐⭐ |
| `per-account-channel-peer` | 按账户 + 渠道 + 发送者隔离 | **多账户推荐** ⭐⭐⭐⭐⭐ |

**安全警告**: 如果 Agent 可以接收来自多个人的 DM，强烈建议启用 `dmScope: per-channel-peer`，否则所有用户共享相同的对话上下文，可能导致隐私泄露。

### Session 维护策略

```json
{
  "mode": "enforce",           // 强制执行清理策略
  "pruneAfter": "45d",         // 清理 45 天前的会话
  "maxEntries": 800,           // 最多 800 个会话
  "rotateBytes": "20mb",       // 超过 20MB 时轮换
  "maxDiskBytes": "1gb",       // 磁盘上限
  "highWaterBytes": "800mb"    // 高水位线（80%）
}
```

## 🧠 三层记忆架构 + WAL Protocol⭐⭐⭐⭐⭐

### 三层记忆系统

```
┌─────────────────────────────────────────────────────────┐
│ Layer 1: 会话记忆（Session Memory）                       │
│ - 当前会话上下文                                        │
│ - 临时决策和中间结果                                    │
└─────────────────────────────────────────────────────────┘
              ↓ 同步关键信息
┌─────────────────────────────────────────────────────────┐
│ Layer 2: 任务记忆（Task Memory）                          │
│ - 任务计划文件                                          │
│ - 子代理执行结果                                        │
└─────────────────────────────────────────────────────────┘
              ↓ 同步重要发现
┌─────────────────────────────────────────────────────────┐
│ Layer 3: 长期记忆（Long-term Memory）                     │
│ - MEMORY.md：精选记忆                                   │
│ - memory/YYYY-MM-DD.md：每日日志                        │
└─────────────────────────────────────────────────────────┘
```

### WAL Protocol（写后读协议）⭐⭐⭐⭐⭐

1. **STOP** — 不要立即回复
2. **WRITE** — 更新记忆文件
3. **THEN** — 回复用户

**触发条件**: 修正、专有名词、偏好、决策、编辑、数值

**最佳实践**:
- DECIDE to write: 决定、偏好、持久事实 → MEMORY.md
- Daily notes: 日常记录 → memory/YYYY-MM-DD.md
- 定期 review: 定期清理 MEMORY.md，移除过时信息
- Ask to remember: 重要事项明确让 Agent 写入记忆
- 立即提交：操作后 `git add` 和 `git commit`

## 🧩 Cron vs Heartbeat（熟练）⭐⭐⭐⭐

### 使用 Heartbeat 当

- ✅ 多个检查可以批量处理（邮件 + 日历 + 通知）
- ✅ 需要对话上下文
- ✅ 时间可以稍有漂移（每~30 分钟）
- ✅ 想通过组合周期性检查减少 API 调用

### 使用 Cron 当

- ✅ 精确时间要求（"每周一 9:00 整"）
- ✅ 任务需要与主会话历史隔离
- ✅ 想要不同的模型或思考级别
- ✅ 一次性提醒（"20 分钟后提醒我"）
- ✅ 输出应直接传递到 channel 而不涉及主会话

**核心原则**: Cron 精确定时，Heartbeat 批量处理，两者互补

## 🧩 Gateway 架构深化（2026-03-23 新增）⭐⭐⭐⭐⭐

### 单一长期运行

一个 Gateway 控制单一 Baileys 会话

### 控制平面

- WebSocket 连接
- Typed API（JSON Schema 验证）

### 事件类型

- `agent` - 代理事件
- `chat` - 聊天事件
- `presence` - 在线状态
- `health` - 健康检查
- `heartbeat` - 心跳检测
- `cron` - 定时任务

### 连接生命周期

```
Client -> Gateway: req:connect
Gateway -> Client: res (ok) + presence + health snapshot
Client -> Gateway: req:agent
Gateway -> Client: event:agent (streaming) + res:agent (final)
```

### 安全机制

- **设备身份**: 所有连接包含 device identity
- **配对流程**: 新设备需要批准，颁发 device token
- **本地连接**: loopback 可自动批准
- **签名验证**: 所有连接必须签名 connect.challenge

### OpenClaw 是什么？（一句话）

> **AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。  
> **它不是聊天机器人**，而是把 AI 连接到真实世界的桥梁。

### 四大核心理念（必背）

1. **Access control before intelligence**（访问控制先于智能）
2. **隐私优先**：私有数据保持私有
3. **记忆即文件**：所有记忆写入 Markdown 文件
4. **工具优先**：第一类工具而非 skill 包裹

### 三层架构（必讲）

```
Agent Layer（智能层）← 大脑
  ↓
Gateway Layer（网关层）← 路由器（不运行 AI，只是调度员）
  ↓
Node Layer（节点层）← 手脚
```

### 四大核心组件

| 组件 | 作用 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI，只是调度员 |
| **Agent** | 执行 AI 任务 | Agent Loop：接收→思考→执行→循环 |
| **Session** | 有状态容器 | 消息历史、上下文、工具状态 |
| **Channel** | 协议适配器 | 支持 15+ 个聊天平台 |

### 三层记忆架构

```
会话记忆 → 任务记忆 → 长期记忆
```

**WAL Protocol**:
- STOP — 不要立即回复
- WRITE — 更新记忆文件
- THEN — 回复用户

### 安全模型

| 级别 | 类型 | 权限范围 |
|------|------|----------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准）|
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |
