## 📚 一、OpenClaw 是什么？

### 1.1 核心定义

OpenClaw 是一个**AI Agent 运行时平台**，更准确地说，它是一个**智能网关（Runtime Gateway）**。

**官方定义**：
- **运行时（Runtime）**：提供 Agent 执行所需的环境、资源和生命周期管理
- **平台（Platform）**：可扩展、可配置的生态系统
- **AI Agent**：以大语言模型（LLM）为核心，具备工具调用能力的智能体

**本质理解**：OpenClaw 是一个**AI 原生时代的运行时基础设施**，是连接用户、AI 模型、工具生态和各类消息平台的智能路由中枢。

### 1.2 核心特点

| 特点 | 说明 |
|------|------|
| **本地优先** | 数据存储在本地，隐私可控 |
| **多渠道接入** | 支持微信、Telegram、Discord、Slack、飞书等 |
| **可扩展** | 通过 Skill 系统扩展功能 |
| **多模型支持** | OpenAI、Anthropic、Google、本地模型等 |

### 2.2 四大核心组件

| 组件 | 职责 | 说明 |
|------|------|------|
| **Gateway** | 中央枢纽 | 生命周期管理、消息路由、工具协调、安全控制、状态持久化 |
| **Agent** | AI 执行体 | 实际执行 AI 任务的实例，有身份、配置、状态、运行时 |
| **Session** | 有状态容器 | 消息历史、上下文窗口、工具状态、元数据 |
| **Channel** | 消息通道 | 协议适配器，支持即时通讯、传统协议、企业平台等 |

### 2.3 Agent Loop 流程

```
1. 接收输入 → 用户通过 Channel 发送消息
2. 构建上下文 → 组装 Session 历史、系统提示词、工具列表
3. LLM 推理 → 模型决定是直接回复还是调用工具
4. 工具执行 → 通过 Gateway 调用外部工具
5. 循环或结束 → 需要多步推理则回到步骤 3
6. 发送响应 → 将 AI 回复通过原 Channel 发送给用户
```

**关键点**：模型拥有决策权，不是被动回答，而是主动决定需要什么信息、调用什么工具。

## 🧠 三、System Prompt：Agent 的"灵魂"

### 3.1 System Prompt 包含内容

- **基础身份**：Agent 的名称、描述、emoji 等
- **工具描述**：当前可用的所有工具及其参数说明（JSON Schema 格式）
- **运行时信息**：当前时间、日期、环境变量等
- **安全提示**：沙箱边界、禁止行为等
- **格式说明**：如何输出工具调用、如何组织回复

### 3.2 动态生成机制

**核心特点**：同一个 Agent，在不同 Session 中可能有不同的工具可用；系统提示词会根据实际情况实时调整。

## 🛠️ 四、Skill 系统与 MCP 协议

### 4.1 MCP（Model Context Protocol）

MCP 是 Anthropic 提出的一种开放标准，**标准化 AI 与外部世界的交互接口**。

**之前的问题**：
- OpenAI 的 Function Calling
- LangChain 的 Tools
- 各平台自定义的插件系统

**MCP 的解决方案**：统一的工具定义方式，一个 MCP 兼容的工具可以被任何支持 MCP 的 AI 系统使用。

### 4.2 OpenClaw Skills

OpenClaw 的 **Skills** 就是 MCP 的实现：
- 每个 Skill 是一个独立的包，包含工具定义和实现
- Skill 通过 JSON Schema 描述工具的输入输出
- Gateway 负责 Skill 的注册、发现和调用
- Agent 通过标准接口与 Skill 交互，无需关心具体实现

**生态效应**：开发者可以编写通用的 Skills，发布到社区，被所有 OpenClaw 用户使用。

### 4.3 Skill 目录结构示例

```
~/.openclaw/skills/task-tracker/
├── SKILL.md          # Skill 定义
├── scripts/
│   ├── create_task.py    # 创建新任务
│   ├── update_task.py    # 更新进度
│   ├── list_tasks.py     # 列出所有任务
│   └── complete_task.py  # 标记完成
└── README.md
```

### 4.4 典型 Skill 案例

#### 案例 1：任务追踪 Skill（Task Tracker）

**解决的问题**：AI 没有持久化的任务记忆，重启后忘记之前做了什么。

**实现方案**：
- 使用 Python 脚本创建/更新/完成/列出任务
- 任务状态保存到 `memory/tasks/ACTIVE-<task-id>.md`
- Markdown 格式，人类和机器都能读
- 启动时自动检查待办

**核心价值**：让 AI 有持久的状态记忆，重启后依然知道要做什么。

#### 案例 2：股票查询 Skill

**解决的问题**：网络不稳定、数据源单一、没有缓存。

**实现方案**：
- 多数据源：腾讯（优先）→ 新浪 → Tushare → Akshare（备选）
- 本地缓存：5 分钟有效期，减少 API 调用
- 自动降级：主数据源失败时切换到备用源

**核心价值**：稳定、快速、可追溯的股票数据查询。

#### 案例 3：自动备份 Skill

**解决的问题**：手动备份容易忘记、备份策略不统一。

**实现方案**：
- Python 脚本自动打包工作空间
- 使用 tar 压缩，排除 `node_modules` 等
- cron 定时任务每 6 小时执行
- 自动清理 7 天前的旧备份

**核心价值**：完全自动化的备份机制，数据零丢失。

## 🗂️ 五、记忆系统架构

### 5.1 三层记忆宫殿

| 层级 | 内容 | 文件位置 | 特点 |
|------|------|----------|------|
| **第一层** | 每日日志（原始记录） | `memory/YYYY-MM-DD.md` | 无限存储，原始内容 |
| **第二层** | 精选记忆（精华提取） | `MEMORY.md` | <2500 字符，自动加载 |
| **第三层** | 长期归档（高价值保存） | `life/` | 决策记录、成就、周报 |

### 5.2 自动化检查点

**机制**：每 6 小时自动从当日日志中提取关键记忆，保存到 `MEMORY.md`。

**脚本逻辑**：
1. 读取今日日志（如最后 150 行）
2. 调用 LLM 分析：今日成就、学到的、重要决策
3. 把提取出的 3-5 个关键点追加到 `MEMORY.md`

**效果**：信息密度从 5% → 85%（17 倍提升！）

## 📊 八、本地模型部署（vLLM）

### 8.1 部署背景

**问题**：阿里云 Coding Plan 的 token 消耗过快，无法支撑日常使用。

**解决方案**：本地部署 vLLM，实现 Token 自由。

### 8.2 硬件配置

- **GPU**: 2 × NVIDIA GeForce RTX 4090 (24GB × 2 = 48GB 总显存)
- **CPU**: 高性能桌面级处理器
- **内存**: 64GB DDR5
- **系统**: Ubuntu 22.04 LTS

### 8.3 模型选择：Qwen3.5-35B-A3B-FP8

**选择原因**：
1. 参数量适中：35B 参数在 2 卡 4090 上可以流畅运行
2. FP8 量化：精度损失小，显存占用大幅降低
3. 性能强劲：代码生成和逻辑推理表现优异
4. 免费开源：可以完全本地化部署

### 8.4 启动脚本

```bash
export VLLM_USE_MODELSCOPE=true

vllm serve Qwen/Qwen3.5-35B-A3B-FP8 \
    --port 8000 \
    --tensor-parallel-size 2 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 200000 \
    --max-num-seqs 2 \
    --enable-prefix-caching \
    --reasoning-parser qwen3 \
    --enable-auto-tool-choice \
    --tool-call-parser qwen3_coder
```

### 8.5 SSH 隧道持久化

**基础命令**：
```bash
ssh -p 6122 -L 8000:localhost:8000 -fN codeserver@39.102.210.43
```

**持久化方案**：
- 隧道管理脚本：自动监控 SSH 隧道状态
- 健康检查脚本：每分钟检查，自动重连
- Cron 定时任务：7×24 小时服务

### 8.6 效果对比

| 指标 | 阿里云 API | 本地 vLLM |
|------|-----------|----------|
| 首字延迟 | 500ms | 200ms |
| 吞吐量 | 50 tokens/s | 120 tokens/s |
| 并发能力 | 受限于 API | 可调整 |
| 数据隐私 | 云端 | 完全本地 |

| 方案 | 每月成本 | Token 限额 |
|------|---------|-----------|
| 阿里云 Coding Plan | $0 (免费) | 约 5 万 tokens |
| 阿里云付费套餐 | ¥200/月 | 约 50 万 tokens |
| **本地 vLLM** | **电费约 ¥50/月** | **无限** |

## 🎯 九、核心经验总结

### 9.1 技能开发经验

1. **保持简单够用** - 先跑通最简单的方案，再考虑优化
2. **文件格式可读** - Markdown/JSON 格式，人类和机器都能读
3. **自动化优于手动** - 能用脚本解决的，绝不用手动操作
4. **多数据源降级** - 防止单一数据源故障导致服务不可用
5. **本地缓存机制** - 减少 API 调用，提升响应速度

### 9.2 记忆系统经验

1. **三层架构** - 原始记录、精选记忆、长期归档，各有所用
2. **自动检查点** - LLM 自动提取关键信息，效率翻倍
3. **定期归档** - 保持 MEMORY.md 精简，控制在 2500 字符内
4. **记录决策** - 不仅记录"做了什么"，还要记录"为什么这么做"

### 9.3 多智能体架构经验

1. **会话间通信需要配置** - 设置 `tools.sessions.visibility=all`
2. **子 agent 的回复机制** - 回复是发送给主 session，不是直接给用户
3. **文档要及时更新** - 更新一个系统后，要系统性地更新所有相关文档
4. **不要过度设计** - 先用最简单的 `subagent` 方式跑通，再考虑优化

### 9.4 本地部署经验

1. **硬件投资是一次性的** - API 调用是持续性的，本地部署长期更划算
2. **SSH 隧道自动重连** - 健康检查脚本是关键，确保服务可用性
3. **多模型部署策略** - 可以根据任务自动切换大小模型
4. **量化选择** - FP8 是性能/精度的平衡点
