### 1️⃣ OpenClaw 定义（精通）⭐⭐⭐⭐⭐

**一句话定义**:
> **OpenClaw 是 AI Agent 运行时平台**，核心是**智能网关（Runtime Gateway）**。  
> **它不是聊天机器人**，而是把 AI 连接到真实世界的桥梁。

**官方描述**:
> OpenClaw 是一个**自托管网关（self-hosted gateway）**，将您最喜欢的聊天应用（WhatsApp、Telegram、Discord、iMessage 等）与 AI 编码智能体连接起来。

**四大核心理念（必背）⭐⭐⭐⭐⭐**:
1. **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
2. **隐私优先**：私有数据保持私有
3. **记忆即文件**：所有记忆写入 Markdown 文件
4. **工具优先**：第一类工具而非 skill 包裹

### 3️⃣ Gateway 架构深度解析（2026-03-26 新增）⭐⭐⭐⭐⭐

#### 核心组件

- **Gateway（守护进程）**:
  - 维护所有 messaging surfaces 的提供者连接
  - 暴露 typed WebSocket API（JSON Schema 验证）
  - 发射事件：`agent`、`chat`、`presence`、`health`、`heartbeat`、`cron`

- **Clients（mac 应用 / CLI / web admin）**:
  - 每个 client 一个 WS 连接
  - 发送请求（`health`、`status`、`send`、`agent`、`system-presence`）
  - 订阅事件（`tick`、`agent`、`presence`、`shutdown`）

- **Nodes（macOS / iOS / Android / headless）**:
  - 连接到**同一个 WS 服务器**，声明 `role: node`
  - 暴露命令：`canvas.*`、`camera.*`、`screen.record`、`location.get`

#### 连接生命周期（single client）

```
Client -> Gateway: req:connect
Gateway -> Client: res (ok) + presence + health snapshot
Gateway -> Client: event:presence
Gateway -> Client: event:tick
Client -> Gateway: req:agent
Gateway -> Client: res:agent {runId, status:"accepted"}
Gateway -> Client: event:agent (streaming)
Gateway -> Client: res:agent {runId, status, summary}
```

#### Wire Protocol（协议细节）

- **传输**: WebSocket, JSON 文本帧
- **第一步**: 必须是 `connect`
- **握手后**:
  - 请求：`{type:"req", id, method, params}` → `{type:"res", id, ok, payload|error}`
  - 事件：`{type:"event", event, payload, seq?, stateVersion?}`
- **认证**: 如果设置了 `OPENCLAW_GATEWAY_TOKEN`，`connect.params.auth.token` 必须匹配
- **幂等性**: 副作用方法（`send`、`agent`）需要幂等键

#### 配对与本地信任

- **设备身份**: 所有 WS 客户端（operators + nodes）在 `connect` 时包含 device identity
- **新设备**: 需要配对批准，Gateway 颁发 **device token**
- **本地连接**: 回环（loopback）或同一台主机的 tailnet 地址可以自动批准
- **签名验证**: 所有 `connect` 必须签名 `connect.challenge` nonce
- **非本地连接**: 仍需要明确批准

#### 远程访问

- **首选**: Tailscale 或 VPN
- **替代**: SSH 隧道
  ```bash
  ssh -N -L 18789:127.0.0.1:18789 user@host
  ```
- **TLS**: 可以在远程设置中启用 WS 的 TLS + pinning

#### 运行时不变性

- 每台主机**恰好一个 Gateway** 控制单个 Baileys 会话
- 握手是强制性的，任何非 JSON 或第一个帧不是 `connect` 会立即关闭
- 事件不可重放，客户端必须在出现间隙时刷新

### 4️⃣ Session 管理深度解析（2026-03-26 新增）⭐⭐⭐⭐⭐

#### Session Key 格式

- **直接聊天**（根据 `session.dmScope`）:
  - `main`: `agent:<agentId>:main`
  - `per-peer`: `agent:<agentId>:direct:<peerId>`
  - `per-channel-peer`: `agent:<agentId>:<channel>:direct:<peerId>`
  - `per-account-channel-peer`: `agent:<agentId>:<channel>:<accountId>:direct:<peerId>`

- **群组聊天**: `agent:<agentId>:<channel>:group:<id>`
- **频道聊天**: `agent:<agentId>:<channel>:channel:<id>`
- **Cron 任务**: `cron:<job.id>` 或 `session:<custom-id>`
- **Webhooks**: `hook:<uuid>`
- **Node 运行**: `node-<nodeId>`

#### 安全 DM 模式（multi-user 场景必配）⭐⭐⭐⭐⭐

**警告**: 如果你的 agent 可以接收**多个人**的 DM，强烈建议启用安全 DM 模式。

**问题示例**:
- Alice (`<SENDER_A>`) 向 agent 发送私密话题（如医疗预约）
- Bob (`<SENDER_B>`) 问 agent "我们刚才在说什么？"
- 因为两个 DM 共享同一个 session，模型可能用 Alice 的上下文回答 Bob

**解决方案**: 设置 `dmScope` 隔离会话

```json5
{
  session: {
    dmScope: "per-channel-peer",  // 多用户推荐
    // 或
    dmScope: "per-account-channel-peer",  // 多账户推荐
  },
}
```

**何时启用**:
- 你有多个配对批准的发件人
- 你使用 DM allowlist 且有多条条目
- 你设置了 `dmPolicy: "open"`
- 多个电话号码或账户可以消息你的 agent

#### Session 维护机制（2026-03-23 新增）

**默认值**:
- `session.maintenance.mode`: `warn`（报告但不清理）
- `session.maintenance.pruneAfter`: `30d`
- `session.maintenance.maxEntries`: `500`
- `session.maintenance.rotateBytes`: `10mb`
- `session.maintenance.maxDiskBytes`: 未设置（禁用）
- `session.maintenance.highWaterBytes`: `maxDiskBytes` 的 80%

**维护流程**（mode: "enforce"）:
1. 清理超过 `pruneAfter` 的陈旧条目
2. 将条目数量限制为 `maxEntries`（ oldest first）
3. 归档不再引用的 transcript 文件
4. 按保留策略清理旧的 `*.deleted.<timestamp>` 和 `*.reset.<timestamp>` 归档
5. `sessions.json` 超过 `rotateBytes` 时轮换
6. 如果设置了 `maxDiskBytes`，强制执行磁盘预算

**性能优化建议**:
- 生产环境使用 `mode: "enforce"` 自动限制增长
- 同时设置时间和数量限制
- 在大型部署中设置 `maxDiskBytes` + `highWaterBytes`
- 运行 `openclaw sessions cleanup --dry-run --json` 验证影响

#### Session 重置策略

- **每日重置**: 默认网关主机当地时间**凌晨 4 点**
- **空闲重置**: `idleMinutes` 增加滑动空闲窗口，两者取先到期者
- **手动重置**:
  - `/new` - 启动新 session id
  - `/reset` - 启动新 session id
  - `/new <model>` - 设置新 session 的 model
  - `/stop` - 终止当前运行
  - `/compact` - 总结旧上下文释放空间

#### Session 状态存储位置

- **Gateway 主机**:
  - 存储文件：`~/.openclaw/agents/<agentId>/sessions/sessions.json`
  - Transcript: `~/.openclaw/agents/<agentId>/sessions/<SessionId>.jsonl`

### 6️⃣ 记忆系统深度解析（精通）⭐⭐⭐⭐⭐

#### 三层记忆架构

```
Layer 1: 会话记忆（Session Memory）
- 当前会话上下文
- 临时决策和中间结果
    ↓ 同步关键信息
Layer 2: 任务记忆（Task Memory）
- 任务计划文件
- 子代理执行结果
    ↓ 同步重要发现
Layer 3: 长期记忆（Long-term Memory）
- MEMORY.md：精选记忆
- memory/YYYY-MM-DD.md：每日日志
```

#### WAL Protocol（写后读协议）

- **STOP** — 不要立即回复
- **WRITE** — 更新记忆文件
- **THEN** — 回复用户

**触发条件**: 修正、专有名词、偏好、决策、编辑、数值

#### Pre-compaction Memory Flush

当 session 接近自动 compaction 时，OpenClaw 可以运行**静默 memory flush**，提醒模型将持久化笔记写入磁盘。这仅在 workspace 可写时运行。

### 7️⃣ Cron vs Heartbeat（深度对比）⭐⭐⭐⭐⭐

#### 使用 Heartbeat 当：
- 多个检查可以批量处理（邮箱 + 日历 + 通知）
- 需要对话上下文
- 时间可以稍有漂移（每~30 分钟）
- 想通过组合周期性检查减少 API 调用

#### 使用 Cron 当：
- 精确时间要求（"每周一上午 9:00 整"）
- 任务需要与主会话历史隔离
- 想要不同的模型或思考级别
- 一次性提醒（"20 分钟后提醒我"）
- 输出应直接传递到 channel 而不涉及主会话

**互补设计**:
- **Cron**: 精确定时，独立执行
- **Heartbeat**: 批量处理，利用对话上下文
