## 什么是 OpenClaw？

### 官方定义

**OpenClaw** 是一个 **AI Agent 运行时平台 (AI Agent Runtime Platform)**。

更准确的理解是：**智能运行时网关 (Smart Runtime Gateway)**。

### 核心定位

```
┌─────────────────────────────────────────────────────────────┐
│                        OpenClaw Gateway                      │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ Telegram │  │ Discord  │  │  Slack   │  │ 其他平台  │    │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘    │
│       └─────────────┴─────────────┴─────────────┘           │
│                         │                                    │
│                    Channel Layer                            │
│                         │                                    │
│              ┌──────────┴──────────┐                        │
│              │   Session Manager   │                        │
│              └──────────┬──────────┘                        │
│                         │                                    │
│              ┌──────────┴──────────┐                        │
│              │     Agent Pool      │                        │
│              └──────────┬──────────┘                        │
│                         │                                    │
│              ┌──────────┴──────────┐                        │
│              │   Tool Registry     │ ← Skills/MCP          │
│              └──────────┬──────────┘                        │
│                         │                                    │
│              ┌──────────┴──────────┐                        │
│              │   LLM Providers     │                        │
│              └─────────────────────┘                        │
└─────────────────────────────────────────────────────────────┘
```

### 核心特点

| 特点 | 说明 |
|------|------|
| **Self-hosted** | 运行在用户自己的硬件上，完全掌控数据隐私 |
| **Multi-channel** | 一个 Gateway 同时服务 WhatsApp、Telegram、Discord、iMessage 等多个平台 |
| **Agent-native** | 为 AI 代码助手设计，支持工具调用、会话管理、记忆系统 |
| **Open source** | MIT 许可，社区驱动 |

### 安装方式

```bash
# 安装 OpenClaw
npm install -g openclaw@latest

# 配置并安装守护进程
openclaw onboard --install-daemon

# 登录渠道并启动 Gateway
openclaw channels login
openclaw gateway --port 18789
```

## 核心架构四大组件

### 1. Gateway (网关)

**职责**：系统的"大脑"和"路由器"

| 职责 | 说明 |
|------|------|
| **生命周期管理** | 启动、停止、监控所有 Agent 实例 |
| **消息路由** | 将消息分发到正确的 Session 和 Agent |
| **工具协调** | 管理 Skill 注册，处理工具调用请求 |
| **安全控制** | 执行沙箱策略，管理权限边界 |
| **状态持久化** | 维护 Session 历史，处理上下文压缩 |

**重要理解**：Gateway **不运行 AI 模型**，它只是 AI 模型的"调度员"。

**优势**：
- **解耦**：Gateway 与 Agent 运行环境分离，可以独立升级
- **安全**：Agent 在隔离环境中运行，即使被攻破也不会影响 Gateway
- **灵活**：可以连接本地模型、远程 API，甚至多个不同的模型提供商

### 2. Agent (AI 执行体)

**定义**：实际执行 AI 任务的实例，有自己的独立环境

每个 Agent 包含：
- **身份 (Identity)**：名称、描述、头像等元信息
- **配置 (Config)**：使用的模型、系统提示词、可用工具等
- **状态 (State)**：当前会话、历史消息、记忆等
- **运行时 (Runtime)**：执行环境（本地进程、Docker、远程等）

**隔离设计**：Agent 运行在隔离环境中，通过 **Bridge Protocol** 与 Gateway 通信。

**原因**：Agent 可能执行不可信的代码（通过工具），必须被限制在沙箱中。

### 3. Session (会话容器)

**定义**：有状态的会话容器，类似 HTTP 是有状态的协议

每个 Session 包含：
- **消息历史**：用户与 AI 的完整对话记录
- **上下文窗口**：当前有效的上下文（经过压缩处理）
- **工具状态**：本次会话中工具调用的中间结果
- **元数据**：创建时间、最后活跃时间、关联的 Channel 等

**核心挑战**：**上下文长度管理**

**解决方案**：**Compaction（压缩）**机制
- 当历史消息超过阈值时，系统智能地总结和裁剪
- 保留关键信息的同时释放空间

### 4. Channel (消息通道)

**定义**：OpenClaw 与外部世界连接的**协议适配器**

**支持的 Channel**：

| 类型 | 平台 |
|------|------|
| **即时通讯** | Telegram、Discord、Slack、WhatsApp、Signal、微信（通过 Lark/Feishu） |
| **传统协议** | IRC、Matrix |
| **企业平台** | Microsoft Teams、Google Chat、飞书 |
| **其他** | iMessage、BlueBubbles、Webhook |

**插件化设计**：每个 Channel 都是一个插件，实现统一接口：
- 接收消息（从平台到 OpenClaw）
- 发送消息（从 OpenClaw 到平台）
- 格式转换（平台特定格式 ↔ OpenClaw 标准格式）

**优势**：新增平台支持只需要开发 Channel 插件，无需改动核心代码。

## Agent 工作原理

### Agent Loop (AI 循环)

```
┌──────────────────────────────────────────────────────────┐
│                      Agent Loop                          │
│                                                          │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│   │ 接收输入 │ → │ 思考决策 │ → │ 执行动作 │             │
│   └─────────┘    └────┬────┘    └────┬────┘             │
│        ↑              │              │                   │
│        │              ↓              ↓                   │
│        │         ┌─────────┐    ┌─────────┐             │
│        │         │ 工具调用 │    │ 直接回复 │             │
│        │         └────┬────┘    └────┬────┘             │
│        │              │              │                   │
│        └──────────────┴──────────────┘                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**具体流程**：

1. **接收输入**：用户通过某个 Channel 发送消息，Gateway 路由到对应 Session 的 Agent
2. **构建上下文**：Gateway 将 Session 历史、系统提示词、可用工具列表组装成完整的 Prompt
3. **LLM 推理**：Agent 调用大模型，模型决定是**直接回复**还是**调用工具**
4. **工具执行**（如果需要）：Agent 通过 Gateway 调用外部工具，获取结果
5. **循环或结束**：如果需要多步推理，回到步骤 3；否则返回最终结果
6. **发送响应**：Gateway 将 AI 的回复通过原 Channel 发送给用户

**关键理解**：**模型拥有决策权**，它不是被动地回答问题，而是主动决定需要什么信息、调用什么工具、如何组织答案。

## Session 与上下文管理

### Context Window (上下文窗口)

**问题**：大模型有 token 限制，无法无限保留历史

**解决方案**：Compaction 机制

**工作流程**：
1. 监控当前 Session 的 token 数量
2. 超过阈值时触发压缩
3. 智能提取关键信息
4. 保留摘要和重要细节
5. 释放空间

### Session Isolation (会话隔离)

每个 Session 完全独立：
- 不同的用户消息 → 不同的 Session
- 同一用户的不同对话 → 不同的 Session
- 会话之间不会互相干扰

**配置**：
```json
{
  "sessions": {
    "visibility": "all"  // 允许跨 agent 访问
  }
}
```

## MCP 协议

### MCP (Model Context Protocol)

**定义**：Anthropic 提出的开放标准，用于统一 AI 与外部世界的交互接口

**核心思想**：标准化 AI 与外部世界的交互接口

### MCP 之前的问题

每个 AI 框架有自己的工具定义方式：
- OpenAI 的 Function Calling
- LangChain 的 Tools
- 各平台自定义的插件系统

### MCP 的优势

**统一标准**：
- 一个 MCP 兼容的工具，可以被任何支持 MCP 的 AI 系统使用
- 开发者编写通用的 Skills，发布到社区
- Gateway 负责 Skill 的注册、发现和调用
- Agent 通过标准接口与 Skill 交互

### JSON Schema 描述

MCP 使用 JSON Schema 描述工具的输入输出：
```json
{
  "type": "object",
  "properties": {
    "title": {"type": "string"},
    "description": {"type": "string"}
  },
  "required": ["title"]
}
```

## 定时任务 (Cron)

### OpenClaw 的定时机制

**核心工具**：Cron (crontab)

Cron 是类 Unix 系统的标准定时任务工具，可以精确控制任务执行时间。

**示例**：
```bash
# 每天早上 8:00 执行
0 8 * * * /path/to/script.sh

# 每 5 分钟执行
*/5 * * * * /path/to/script.sh

# 每周一 22:00 执行
0 22 * * 1 /path/to/script.sh
```

### Cron 配置示例

```json
{
  "id": "memory-checkpoint",
  "name": "记忆检查点",
  "description": "每 6 小时从每日日志中提取关键记忆",
  "schedule": {
    "kind": "cron",
    "expr": "0 */6 * * *",
    "tz": "UTC"
  },
  "command": [
    "bash",
    "/home/claw/.openclaw/workspace/para-system/checkpoint-memory-llm.sh"
  ]
}
```

### 常见问题

**问题 1：Cron 环境差异**
- Cron 执行时的环境变量和交互式 shell 不同
- **解决**：使用绝对路径，显式加载环境

**问题 2：时区问题**
- Cron 使用系统时区，可能不是用户所在地时区
- **解决**：在脚本中显式指定时区

**问题 3：权限问题**
- Cron 任务可能没有权限访问某些文件
- **解决**：确保脚本和配置文件有正确的权限

## 记忆系统架构

### 三层记忆宫殿

#### 第一层：每日日志（原始记录）

**位置**：`~/.openclaw/workspace/memory/YYYY-MM-DD.md`

- 无限存储，原始内容
- 不删不减，所有细节都记下来
- 像写日记一样记录每天的工作

**示例**：
```markdown
# 2026-03-10

## 今日工作
- 学习了 OpenClaw 核心架构
- 整理了两篇博客文章
- 创建了任务追踪 Skill

## 遇到问题

- 通过配置 tools.sessions.visibility=all 解决

## 收获
- 理解了 Gateway 和 Agent 的分工
- 掌握了 Cron 定时任务的配置方法
```

#### 第二层：精选记忆（精华提取）

**位置**：`~/.openclaw/workspace/MEMORY.md`

**特点**：
- 从每日日志中提取的精华
- 每次打开都自动加载
- 控制在 2500 字符以内
- 包含系统配置、重要决策、个人信息

**示例内容**：
- 系统配置
- 重要决策
- 近期成果
- 个人信息

#### 第三层：长期归档（高价值保存）

**位置**：`~/.openclaw/workspace/life/`

**结构**：
```
life/
├── decisions/       # 决策记录
├── motivation/      # 成就和连胜
└── archives/        # 周报和归档
    └── weekly/
```

**用途**：
- 为什么做某个决定（决策日志）
- 完成了什么成就（动机系统）
- 每周总结（周报）

### 自动化检查点

**脚本**：`checkpoint-memory-llm.sh`

**功能**：
1. 读取今天的日志（比如最后 150 行）
2. 调用 LLM 分析：
   - 今日成就是什么？

3. 把提取出的 3-5 个关键点追加到 `MEMORY.md`

**压缩率**：从 150 行 → 5 条，**压缩率 97%**！

**定时任务**：每 6 小时自动运行一次

## 最佳实践与踩坑记录

### 踩坑 1：Session 间通信限制

**问题**：
```
Session send visibility is restricted.
Set tools.sessions.visibility=all to allow cross-agent access.
```

**原因**：未配置跨 agent 访问权限

**解决**：
```bash
# 在 openclaw.json 中添加
{
  "tools": {
    "sessions": {
      "visibility": "all"
    }
  }
}
```

**重启 Gateway**：
```bash
openclaw gateway restart
```

---

### 踩坑 2：ACP runtime 未配置

**问题**：
```
Error: ACP runtime backend is not configured. 
Install and enable the acpx runtime plugin.
```

**原因**：ACP runtime 后端未配置

**解决**：改用 `runtime: "subagent"` 方式！

```python
sessions_spawn(
    agentId="code-executor",
    runtime="subagent",  # 改用 subagent
    mode="run",
    task="..."
)
```

---

### 踩坑 3：子 agent 完成后不汇报

**问题**：子 agent 执行完任务后，结果没有自动汇报

**原因**：子 agent 的回复发送给**主 session** (main)，而不是直接发送给最终用户

**解决**：修改 main 会话的 SOUL.md，内置监控逻辑

**流程**：
1. 主会话接收任务
2. 分派给子 agent
3. 子 agent 完成任务，发送回 main
4. main 会话格式化结果，汇报给用户

---

### 踩坑 4：文档更新不及时

**问题**：更新一个系统后，只更新了 MEMORY.md，忘记其他文档

**解决方案**：系统性地更新所有相关文档

**检查清单**：
- [ ] `memory/YYYY-MM-DD.md` - 当天的完整记录
- [ ] `MEMORY.md` - 精选记忆更新
- [ ] `SOUL.md` - 核心身份和职责
- [ ] `IDENTITY.md` - 系统状态更新
- [ ] `TOOLS.md` - 本地配置更新
- [ ] `USER.md` - 用户信息更新

---

### 踩坑 5：Cron 时区问题

**问题**：Cron 使用系统时区，用户在其他时区

**解决**：在脚本中显式指定时区

```python
from datetime import datetime, timezone
tz = timezone(timedelta(hours=8))  # 北京时间
```
