## 🎯 核心知识点总结

### 一、OpenClaw 是什么？

**一句话介绍**:
> **OpenClaw 是一个自托管的 AI Agent 运行时平台**，不是聊天机器人，而是把 AI 模型连接到真实世界的桥梁。

**四大核心理念**:
1. ✅ **Access control before intelligence**（访问控制先于智能）⭐⭐⭐⭐⭐
2. ✅ **隐私优先**：私有数据保持私有
3. ✅ **记忆即文件**：所有记忆写入 Markdown 文件
4. ✅ **工具优先**：第一类工具而非 skill 包裹

**不是什么**:
- ❌ 不是聊天机器人（如 ChatGPT）
- ❌ 不是多租户共享 AI 服务
- ❌ 不是云端托管的 SaaS 产品
- ❌ 不是单一 AI 模型的替代品

**是什么**:
- ✅ **自托管平台**: 运行在自己的设备上，数据本地化
- ✅ **单一控制平面**: Gateway 统一管理多个渠道
- ✅ **Agent 原生**: 内置工具调用、会话管理、多智能体路由
- ✅ **开源项目**: MIT 许可，社区驱动

### 三、四大核心组件

| 组件 | 职责 | 关键点 |
|------|------|--------|
| **Gateway** | 大脑、路由器 | 不运行 AI 模型，只是调度员 |
| **Agent** | 执行 AI 任务 | 身份 + 配置 + 状态 + 运行时 |
| **Session** | 有状态容器 | 消息历史、上下文、工具状态 |
| **Channel** | 协议适配器 | Telegram、Discord、飞书等 |

**Agent Loop 核心循环**:
1. 接收输入 → 2. 思考决策 → 3. 执行动作 → 4. 循环或发送响应

### 五、三层记忆架构

```
┌─────────────────────────────────────────────────────────────┐
│ Layer 1: 会话记忆（Session Memory）                          │
│ - 当前会话上下文                                             │
│ - 临时决策和中间结果                                         │
└─────────────────────────────────────────────────────────────┘
         │ 同步关键信息
         ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 2: 任务记忆（Task Memory）                             │
│ - 任务计划文件                                               │
│ - 子代理执行结果                                             │
└─────────────────────────────────────────────────────────────┘
         │ 同步重要发现
         ▼
┌─────────────────────────────────────────────────────────────┐
│ Layer 3: 长期记忆（Long-term Memory）                        │
│ - MEMORY.md: 精选记忆                                       │
│ - memory/YYYY-MM-DD.md: 每日日志                            │
└─────────────────────────────────────────────────────────────┘
```

**WAL Protocol（写入前置日志协议）**:
- **规则**: 在回复之前，先把关键信息写入 `SESSION-STATE.md`
- **触发条件**:
  - ✏️ **修正**: "It's X, not Y" / "Actually..." / "No, I meant..."
  - 📍 **专有名词**: 人名、地名、公司名、产品名
  - 🎨 **偏好**: 颜色、风格、方法、"我喜欢/不喜欢"
  - 📋 **决策**: "Let's do X" / "Go with Y" / "Use Z"
  - 🔢 **数值**: 数字、日期、ID、URL

### 六、安全模型（重点）

#### 权限层级

| 级别 | 类型 | 权限范围 |
|------|------|----------|
| Level 5 | 主 Agent | 完全权限 |
| Level 4 | 可信子 Agent | 受限系统权限（需批准） |
| Level 3 | 标准子 Agent | 标准开发权限 |
| Level 2 | 受限子 Agent | 严格受限权限 |
| Level 1 | 只读子 Agent | 只读访问 |

#### 信任边界

| 边界或控制 | 含义 | 常见误解 |
|----------|------|---------|
| `gateway.auth` | 认证调用者到 gateway APIs | "需要每个帧的签名" |
| `sessionKey` | 路由键用于上下文/会话选择 | "sessionKey 是用户认证边界" |
| Prompt/content guardrails | 减少模型滥用风险 | "prompt injection 等于认证绕过" |
| `canvas.eval` / browser evaluate | 启用的 operator 能力 | "任何 JS eval 都是漏洞" |
| Node pairing | 配对设备的远程执行 | "设备控制应视为未授权访问" |

#### 安全审计命令

```bash
openclaw security audit       # 基本检查
openclaw security audit --deep # 深度检查
openclaw security audit --fix  # 自动修复
openclaw security audit --json # JSON 格式输出
```

### 九、部署选项

#### 1. 本地部署（推荐）
- Gateway 运行在本地机器
- 所有数据本地化
- 最简单的配置
- 需要本地有 Node.js 环境

#### 2. 远程部署（VPS）
- Gateway 运行在 VPS
- 通过 SSH 隧道或 Tailscale 访问
- 适合需要 24/7 运行的场景
- 可以配合本地节点使用

#### 3. Docker 部署

```bash
docker run -d \
  -v ~/.openclaw:/root/.openclaw \
  -p 18789:18789 \
  openclaw/openclaw
```

#### 系统要求
- **Node.js**: >= 22（必须）
- **包管理器**: `pnpm` 推荐（也可以使用 npm、bun）
- **OS**: macOS、Linux、Windows (via WSL2)
- **资源**: 最低 512MB RAM、1 核心、500MB 磁盘
