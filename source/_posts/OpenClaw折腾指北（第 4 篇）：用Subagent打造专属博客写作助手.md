在前几期的OpenClaw折腾指北中，我们已经介绍了如何安装配置OpenClaw、连接各种消息平台，以及基础的使用方法。今天这一期，我想聊聊一个更进阶的话题——**Subagent（子智能体）**，以及如何用它来打造一个专门帮你写博客的Agent。

<!-- more -->

## 什么是Subagent？

简单来说，Subagent就是OpenClaw中的一个"分身"功能。你可以把它理解为：主Agent（也就是你平时对话的那个AI）可以"召唤"出一个专门处理特定任务的子Agent，让它独立去完成某项工作。

这个子Agent：
- 有独立的对话上下文，不会污染主会话
- 可以配置不同的模型参数（比如用更强的模型处理复杂任务）
- 可以访问特定的工具和技能
- 任务完成后会把结果汇报给主Agent

## 为什么要用Subagent写博客？

你可能会问：直接用主Agent写不就行了，为什么要搞这么复杂？

好问题。让我用实际体验来回答：

**1. 上下文隔离**

写博客往往需要多轮迭代：改标题、调结构、润色段落、加例子...这些中间过程如果都在主会话里，会让对话历史变得臃肿，影响后续其他任务的效率。

**2. 专业化配置**

博客写作Agent可以预先配置好：
- 你的博客主题和风格偏好
- 常用的标签和分类
- 特定的写作模板
- 甚至连接Hexo的发布工具

**3. 并行处理**

你可以同时开启多个Subagent处理不同的博客主题，而主会话保持清爽，随时响应你的其他需求。

**4. 可复用性**

一旦配置好，这个博客写作Agent可以随时被召唤，保持一致的输出质量。

## 创建博客写作Agent的两种方式

OpenClaw提供了两种创建Subagent的方式，各有适用场景。

### 方式一：自然语言命令（推荐新手）

这是最简单直接的方式，适合快速验证想法或临时任务。

**基本语法：**

```
/new agent:博客写手
你是一个专业的技术博客写手，擅长写OpenClaw相关的教程文章。
你的任务是：
1. 根据我提供的主题和大纲撰写文章
2. 使用Markdown格式
3. 语言风格要轻松但专业
4. 文章结构清晰，有适当的代码示例

完成后请返回完整的文章内容。
```

**实际使用示例：**

假设我现在想写一篇关于"OpenClaw连接飞书"的文章，我可以这样调用：

```
@博客写手 帮我写一篇关于"OpenClaw连接飞书机器人"的教程，要求：
- 包含完整的配置步骤
- 有截图说明的位置标记
- 最后要有常见问题排查
- 字数控制在1500字左右
```

Subagent会独立处理这个任务，完成后返回文章正文。主会话保持干净，我只拿到最终结果。

**优点：**
- 零配置，即用即走
- 适合一次性或试验性任务
- 指令可以随时调整

**缺点：**
- 每次都要重新定义角色
- 无法持久化配置
- 复杂的系统提示词会比较冗长

### 方式二：手动创建配置文件（推荐长期使用）

如果你发现某个Subagent的使用频率很高，就应该考虑把它配置成一个"常驻"Agent。

**步骤1：创建Agent配置文件**

在OpenClaw的配置目录下创建新的Agent定义文件：

```bash
mkdir -p ~/.openclaw/agents
vim ~/.openclaw/agents/blog-writer.yaml
```

**步骤2：编写配置**

```yaml
id: blog-writer
name: 博客写手
description: 专业的技术博客写作助手，擅长Markdown格式和教程类文章

# 系统提示词
system_prompt: |
  你是一个专业的技术博客写手，专注于OpenClaw和AI工具相关的教程文章。
  
  ## 写作风格
  - 语言轻松但不失专业
  - 善用类比和比喻解释复杂概念
  - 适当使用emoji增加可读性
  - 代码示例要完整且可运行
  
  ## 文章结构
  1. 引人入胜的开头（问题或场景引入）
  2. 清晰的步骤说明
  3. 必要的截图/代码位置标记
  4. 常见问题或进阶提示
  5. 简洁的结尾
  
  ## 格式要求
  - 使用Markdown格式
  - 代码块标注语言类型
  - 关键步骤用加粗或列表突出
  - 适当使用引用块和分割线

# 可用工具（可选）
tools:
  - file_read
  - file_write
  - web_search

# 模型配置（可选）
model: default
thinking: false
```

**步骤3：注册Agent**

编辑主配置文件 `~/.openclaw/config.yaml`，在 `agents` 部分添加：

```yaml
agents:
  - id: blog-writer
    enabled: true
```

或者使用命令行：

```bash
openclaw agent add blog-writer
```

**步骤4：使用**

现在你可以随时这样调用：

```
@博客写手 我想写一篇关于OpenClaw定时任务的文章
```

**优点：**
- 配置一次，永久使用
- 可以精细控制可用工具和模型
- 支持版本管理（配置文件可以git管理）
- 可以在团队间共享

**缺点：**
- 需要一定的配置工作
- 修改后需要重启或重载配置

## 实际效果展示

让我用刚刚创建的博客写手Agent来演示一下实际效果。

**我的指令：**

```
@博客写手 帮我写一篇关于"为什么要用Subagent"的短文，
面向已经对OpenClaw有基础了解的用户，
重点讲清楚使用场景和实际好处，
500字左右。
```

**Agent返回的内容（节选）：**

> 想象你正在用OpenClaw处理一堆杂事：查邮件、看日历、写代码、还要准备一篇博客...
>
> 如果所有事情都在一个对话里完成，上下文会乱成一锅粥。更重要的是，写博客需要一种"创作模式"的思维状态，而查邮件是"处理模式"——强行混在一起，两边都做不好。
>
> Subagent就是解决这个问题的钥匙。
>
> 它让你可以"分身"出一个专门写博客的助手，这个助手：
> - 只关注写作任务，不会被其他消息打断
> - 可以记住你的风格偏好（通过系统提示词预设）
> - 完成后把结果交给你，不留下任何"创作过程中的废话"
>
> 这就像是你请了一个专门的撰稿人，而你只需要告诉他写什么，最后验收成果。

可以看到，Agent很好地理解了任务要求，输出了一篇结构清晰、语言流畅的短文。

## Subagent的更多可能性

博客写作只是Subagent的一个应用场景。实际上，你可以为任何重复性、专业性的任务创建专门的Agent：

| 场景 | Agent角色 | 功能 |
|------|----------|------|
| 代码审查 | Code Reviewer | 检查代码风格、潜在bug、优化建议 |
| 翻译助手 | Translator | 中英互译，保持技术术语准确 |
| 数据分析师 | Data Analyst | 处理CSV/JSON数据，生成图表建议 |
| 邮件起草 | Email Writer | 根据要点起草正式/商务邮件 |
| 学习导师 | Learning Coach | 解释概念、出题测试、制定学习计划 |

## 总结

Subagent是OpenClaw中一个非常强大的功能，它让AI助手从"一个万能但杂乱的对话"变成了"多个专业且专注的助手团队"。

对于博客写作这种需要一定创作流程的任务，Subagent的优势尤为明显：

1. **上下文隔离** - 创作过程不污染主会话
2. **专业化** - 可以针对写作场景优化提示词和工具
3. **可复用** - 配置一次，随时召唤
4. **可扩展** - 从简单命令到复杂配置，按需选择

如果你经常需要AI辅助写作，强烈建议尝试创建一个专门的博客写作Agent。一开始可以用自然语言命令快速验证，用得顺手了再做成配置文件，长期受益。
