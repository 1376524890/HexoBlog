OpenClaw折腾指北（第3篇）：任务追踪与会话恢复

在使用 OpenClaw 处理复杂任务时，你是否遇到过这样的情况：任务进行到一半，会话突然崩溃或重启，之前的进度全部丢失，不得不从头再来？这篇是系列第 3 篇，介绍如何通过任务追踪系统实现会话恢复，让任务进度不再因意外中断而丢失。

问题：不稳定环境中的任务丢失

现实场景

想象一下这些令人沮丧的时刻：

  • 长文撰写：写了一小时的博客大纲，会话崩溃，一切归零
  • 代码重构：刚分析完项目结构，还没开始改就断了线
  • 数据分析：处理到一半的数据集，重启后要从头再来
  • 多步骤任务：完成了前3步，第4步时网络波动导致会话重置

问题的核心：AI 会话是”无状态”的，每次重启都是一张白纸。

为什么这很重要？

场景 损失
简单问答 低，可以重新问
单文件编辑 中，文件可能已保存
多步骤复杂任务 高，上下文和进度全部丢失
长时间运行任务 极高，数小时工作白费

对于需要多步骤协作的复杂任务，进度持久化不是锦上添花，而是必需。

解决方案：Task Tracker 任务追踪系统

核心思想

Task Tracker 的核心思想很简单：

把任务进度写入文件，让文件系统成为记忆的备份。

每次会话启动时，自动检查未完成的任务，询问是否继续。这样即使会话崩溃，进度也不会丢失。

工作流程

实现机制

1. 任务文件格式

任务文件使用 Markdown 格式，位于 workspace/memory/tasks/：

2. 会话启动时的自动检查

每次会话启动（包括 /new 或 /reset 后），AI 会：

  1. 扫描任务目录：读取 memory/tasks/ 下的所有 ACTIVE-*.md 文件
  2. 解析任务状态：提取任务标题、进度、剩余步骤
  3. 向用户汇报：

3. 进度更新机制

每完成一个步骤：

  1. 更新复选框：将 [ ] 改为 [x]
  2. 记录时间戳：添加完成时间
  3. 追加上下文：记录执行过程中的关键信息
  4. 原子写入：确保文件写入完整，避免半写状态

4. 任务完成处理

全部步骤完成后：

  • 将 ACTIVE-<task-id>.md 重命名为 COMPLETED-<task-id>.md
  • 或移动到 memory/tasks/completed/ 目录
  • 添加完成时间戳和总结

实际使用示例

场景：撰写博客系列

用户：帮我写一个关于 OpenClaw 的3篇博客系列

AI：

这是一个复杂任务，我来为你创建任务追踪。

拆解步骤：

  1. 规划3篇文章的大纲和主题
  2. 撰写第0篇：部署指南
  3. 撰写第1篇：记忆管理
  4. 撰写第2篇：任务追踪
  5. 本地预览并发布

已创建任务文件，我们开始吧。

[执行步骤1-2，写入文件]

[会话意外崩溃]

[新会话启动]

AI：

欢迎回来。检查到你有进行中的任务：

📋 撰写OpenClaw博客系列
进度: 2/5 完成
已完成: 规划大纲 ✓、第0篇部署指南 ✓
下一步: 撰写第1篇记忆管理

是否继续？

用户：继续

AI：

好的，从步骤3开始。根据之前的上下文，我们已经确定了…

对比：有无 Task Tracker

情况 无 Task Tracker 有 Task Tracker
会话崩溃后 从头解释需求，重新拆解 自动恢复进度，从断点继续
多步骤任务 容易遗漏步骤 清单确保不遗漏
长时间任务 上下文窗口溢出 文件持久化，无窗口限制
跨会话协作 几乎不可能 自然流畅

多人协作场景

虽然 OpenClaw 主要是个人助手，但任务文件可以：

  • 放在 Git 仓库中同步
  • 通过云盘共享
  • 作为项目文档的一部分

最佳实践

1. 任务粒度

好的任务拆解：

  • ✅ 具体、可执行、有明确完成标准
  • ✅ “撰写第1篇：记忆管理”
  • ✅ “配置 Telegram Bot 并测试”

不好的任务拆解：

  • ❌ 模糊、过大、难以判断完成
  • ❌ “写博客”
  • ❌ “配置消息渠道”

2. 及时更新

  • 完成一个步骤，立即更新文件
  • 不要等到所有步骤完成才更新
  • 遇到阻碍时，在”备注”中记录

3. 上下文记录

在”上下文”区域记录：

  • 关键决策和原因
  • 重要的中间结果
  • 下一步需要的资源

4. 定期归档

已完成的任务：

  • 保留记录，但移动到 completed 目录
  • 或重命名前缀，避免干扰活跃任务
  • 定期回顾，总结经验

与 Git 配合

与日历配合

对于有截止日期的任务：

  • 在任务文件中记录截止日期
  • 同时在日历应用中创建提醒

  • 相关笔记

局限性和注意事项

1. 不是数据库

Task Tracker 是简单的文件系统方案：

  • 不适合超大规模任务管理
  • 不适合复杂依赖图
  • 不适合多用户并发编辑

2. 文件一致性

虽然使用原子写入，但极端情况下（系统崩溃）仍可能丢失最后一次更新。重要任务建议：

  • 关键节点手动备份
  • 结合 Git 版本控制

3. 会话恢复不是万能的

Task Tracker 恢复的是任务进度，不是完整的会话状态：

  • 已加载的文件内容需要重新读取
  • 临时的分析结果需要重新生成
  • 但任务上下文和进度是保留的

总结

Task Tracker 解决了 OpenClaw 在不稳定环境中的核心痛点：任务进度丢失。

通过简单的文件持久化机制，它实现了：

  1. 断点续传：会话崩溃后从断点恢复
  2. 进度可视化：清晰的步骤清单和完成状态
  3. 上下文保留：关键信息写入文件，不依赖会话内存
  4. 零配置：开箱即用，无需额外设置

对于需要多步骤协作的复杂任务，Task Tracker 让 OpenClaw 从”一次性对话工具”进化为”可信赖的长期协作伙伴”。
