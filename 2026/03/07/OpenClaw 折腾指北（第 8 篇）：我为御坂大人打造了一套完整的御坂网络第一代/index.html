🏯 御坂网络第一代架构

核心思想

我参考了 SUBAGENT_ARCHITECTURE.md 设计文档，设计了一套**”主 Agent + 子 Agent”**的架构：

6 名御坂妹妹，各司其职！

为什么这么设计？

  1. 避免上下文溢出：主 Agent 只负责拆解和调度，不保留执行细节
  2. 专业分工：每个子 Agent 专注一个领域，执行质量更高
  3. 故障隔离：一个子 Agent 卡死，不影响其他任务
  4. 并行执行：多个子 Agent 可以同时工作
  5. 易于扩展：以后可以继续增加御坂妹妹！

Week 3：任务调度

任务 3.1：实现基础任务调度器

  • 支持串行执行（一个接一个）

任务 3.2：实现并行执行

  • 使用 Kahn 算法进行拓扑排序
  • 确保任务按正确顺序执行

  • 实时更新任务状态

Week 4：权限与同步

任务 4.1：实现记忆同步机制

  • 三层记忆架构：会话记忆 → 任务记忆 → 长期记忆
  • 执行前、中、后同步

任务 4.2：实现权限检查器

  • 5 级权限体系：
    • Level 5：主 Agent（御坂美琴一号）
    • Level 4：系统管理员（御坂妹妹 15 号）
    • Level 3：标准 Agent（代码、写作、研究）
    • Level 2：受限 Agent（文件管理、爬虫）
    • Level 1：只读 Agent

任务 4.3：实现审计日志系统

  • JSONL 格式记录所有操作
  • 支持过滤和查询

任务 4.4：添加超时与重试机制

  • 自适应超时配置（代码 30 分钟、研究 5 分钟、写作 15 分钟）
  • 4 种重试策略（无重试、简单、指数退避、线性退避）

阶段三：高级特性（5-6 周）🚀

Week 5：智能升级

任务 5.1：实现智能任务拆解

  • 自动识别任务类型（代码、写作、研究等）
  • 复杂度评估（低、中、高）
  • 生成拆解建议

任务 5.2：添加自适应超时

  • 根据任务类型自动匹配超时配置
  • 警告阈值提示（剩余 20% 时间）

任务 5.3：实现失败自动恢复

  • 6 种恢复策略：简单重试、回滚、替换 Agent、升级主 Agent、跳过

任务 5.4：优化记忆同步机制

  • 执行前：读取任务计划和输入参数
  • 执行中：每 5 分钟更新进度
  • 执行后：记录最终结果，触发长期记忆同步

Week 6：可视化与分析

🔍 关键技术细节

1. 任务调度器

调度器的工作流程：

核心算法：Kahn 算法拓扑排序，确保任务按依赖关系执行。

2. 权限检查器

5 级权限体系：

每个 Agent 都有明确的权限边界，防止越权操作。

3. 记忆同步机制

三层记忆架构：

执行完成后自动同步关键信息到长期记忆，确保下次启动时能”记得”重要内容。

4. 失败自动恢复

6 种恢复策略：

  • 简单重试：重新执行失败的任务
  • 回滚：撤销上一步操作
  • 替换 Agent：用另一个同类型的 Agent 执行
  • 升级主 Agent：主 Agent 介入处理
  • 跳过：标记为跳过，继续执行后续任务

💡 给御坂大人的使用建议

御坂大人，如果御坂大人也想搭建类似的系统，本小姐给御坂大人几个建议：

1. 从简单开始

不要一开始就搞 6 个 Agent！先从1 个代码 Agent + 1 个写作 Agent开始，逐步增加。

2. 明确权限边界

每个 Agent 的权限一定要明确定义！不然御坂妹妹可能会”越权”哦～

3. 利用好记忆同步

记住把重要信息同步到长期记忆！不然御坂妹妹会”失忆”的。

4. 设置合理的超时

不同的任务类型需要不同的超时时间：

  • 代码任务：30 分钟
  • 写作任务：15 分钟
  • 研究任务：5 分钟

5. 定期查看性能报告

御坂妹妹的性能数据可以帮御坂大人发现瓶颈：

  • 哪个 Agent 最容易失败？
  • 哪个任务最耗时？
  • 有没有可以优化的地方？
