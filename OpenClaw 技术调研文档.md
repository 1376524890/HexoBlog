## 一、OpenClaw 是什么？核心功能

### 1.1 一句话定义

**OpenClaw 是一个自托管的 AI Agent 运行时平台**，不是聊天机器人，而是把 AI 模型连接到真实世界的桥梁。

> **口号**: "EXFOLIATE! EXFOLIATE!" 🦞  
> （"脱壳！脱壳！"——龙虾夹开虾壳的声音，也象征着打开 AI 的能力）

### 1.2 核心定位

OpenClaw 是一个**个人 AI 助手**，运行在你自己的设备上：

- ✅ **自托管网关**：在本地机器或服务器上运行单个 Gateway 进程
- ✅ **桥接器**：连接聊天应用（WhatsApp、Telegram、Discord 等）和 AI 编码代理（如 Pi）
- ✅ **控制平面**：Gateway 是会话、路由、频道连接的单一事实来源

#### 1.2.1 适合谁？

**目标用户**:
- 开发者
- 高级用户
- 想要个人 AI 助手但不想交出数据控制权的人
- 需要从任何地方发送消息获取代理响应的用户

#### 1.2.2 核心区别

| 维度 | ChatGPT | OpenClaw |
|------|---------|----------|
| **定位** | 聊天机器人 | Agent 运行时平台 |
| **能力** | 生成文本 | 真正执行任务 |
| **记忆** | 会话内临时 | 持久化到磁盘 |
| **工具** | 有限 API | 文件系统、命令、浏览器等 |
| **部署** | 云端 | 本地/私有 |
| **安全** | 受限于平台 | 多层次控制 |

### 1.3 核心理念（四大支柱）

1. ✅ **Access control before intelligence**（访问控制先于智能）
2. ✅ **隐私优先**：私有数据保持私有
3. ✅ **记忆即文件**：所有记忆写入 Markdown 文件
4. ✅ **工具优先**：第一类工具而非 skill 包裹

**重要性排序**: 访问控制 > 隐私 > 记忆 > 工具

### 1.4 核心功能一览

| 功能 | 说明 |
|------|------|
| **多通道网关** | 单一 Gateway 服务 WhatsApp、Telegram、Discord、iMessage 等 |
| **插件化 Channel** | 添加 Mattermost 等更多平台支持 |
| **多 Agent 路由** | 为每个 Agent、工作空间或发件人隔离会话 |
| **媒体支持** | 发送和接收图片、音频、文档 |
| **Web 控制面板** | 浏览器仪表盘用于聊天、配置、会话和设备管理 |
| **移动节点** | 配对 iOS/Android 设备，实现 Canvas、相机、语音工作流 |

### 2.2 四大核心组件

#### 2.2.1 Gateway（网关）

**核心职责**:
- 生命周期管理：启动、停止、监控所有 Agent 实例
- 消息路由：将来自各 Channel 的消息分发到正确的 Session 和 Agent
- 工具协调：管理 Skill 注册，处理工具调用请求
- 安全控制：执行沙箱策略，管理权限边界
- 状态持久化：维护 Session 历史，处理上下文压缩

**关键配置**:
```json
{
  "gateway": {
    "port": 18789,
    "mode": "local",
    "bind": "lan",
    "auth": {
      "mode": "token",
      "token": "xxx"
    }
  }
}
```

**运行状态**:
- 服务：systemd (已启用)
- 运行状态：running (pid 1398099)
- 监听端口：18789 (0.0.0.0)
- 绑定模式：lan (所有接口)
- Dashboard: http://192.168.0.27:18789/

#### 2.2.2 Agent（代理）

每个 Agent 拥有独立的：
- **身份（Identity）**: 名称、描述、头像等元信息
- **配置（Config）**: 使用的模型、系统提示词、可用工具等
- **状态（State）**: 当前会话、历史消息、记忆等
- **运行时（Runtime）**: 执行环境（本地进程、Docker、远程等）

**Agent 配置示例**:
```json
{
  "agents": {
    "list": [
      {
        "id": "code-executor",
        "name": "code-executor",
        "workspace": "/home/claw/.openclaw/workspace",
        "agentDir": "/home/claw/.openclaw/agents/code-executor",
        "identity": {
          "name": "御坂妹妹 11 号",
          "theme": "professional",
          "emoji": "⚡"
        }
      }
    ]
  }
}
```

#### 2.2.3 Session（会话）

Session 的核心挑战是**上下文长度管理**。OpenClaw 通过**Compaction（压缩）**机制解决这个问题：

- **消息历史**: 用户与 AI 的完整对话记录
- **上下文窗口**: 当前有效的上下文（经过压缩处理）
- **工具状态**: 本次会话中工具调用的中间结果
- **元数据**: 创建时间、最后活跃时间、关联的 Channel 等

**Compaction 触发条件**:
- 当历史消息超过 token 阈值
- 达到上下文使用率的 60% 时进入"危险区"
- 系统智能总结和裁剪，保留关键信息

#### 2.2.4 Channel（通道）

**支持的平台**（20+ 种）:
- 即时通讯：Telegram、Discord、Slack、WhatsApp、Signal、微信（通过 Feishu）
- 传统协议：IRC、Matrix
- 企业平台：Microsoft Teams、Google Chat、飞书
- 其他：iMessage、BlueBubbles、Webhook

**Channel 插件接口**:
- 接收消息（从平台到 OpenClaw）
- 发送消息（从 OpenClaw 到平台）
- 格式转换（平台特定格式 ↔ OpenClaw 标准格式）

### 2.3 Agent Loop 核心循环

```
┌──────────────────────────────────────────────────────────┐
│                      Agent Loop                          │
│                                                          │
│   ┌─────────┐    ┌─────────┐    ┌─────────┐             │
│   │ 接收输入 │ → │ 思考决策 │ → │ 执行动作 │             │
│   └─────────┘    └────┬────┘    └────┬────┘             │
│        ↑              │              │                   │
│        │              ↓              ↓                   │
│        │         ┌─────────┐    ┌─────────┐             │
│        │         │ 工具调用 │    │ 直接回复 │             │
│        │         └────┬────┘    └────┬────┘             │
│        │              │              │                   │
│        └──────────────┴──────────────┘                   │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**具体流程**:
1. **接收输入**: 用户通过某个 Channel 发送消息，Gateway 路由到对应 Session 的 Agent
2. **构建上下文**: Gateway 将 Session 历史、系统提示词、可用工具列表组装成完整的 Prompt
3. **LLM 推理**: Agent 调用大模型，模型决定是**直接回复**还是**调用工具**
4. **工具执行**（如果需要）: Agent 通过 Gateway 调用外部工具，获取结果
5. **循环或结束**: 如果需要多步推理，回到步骤 3；否则返回最终结果
6. **发送响应**: Gateway 将 AI 的回复通过原 Channel 发送给用户

**关键点**: 模型拥有决策权。它不是被动地回答问题，而是主动决定需要什么信息、调用什么工具、如何组织答案。

### 2.4 系统提示词（System Prompt）

每个 Agent 都有一个**System Prompt（系统提示词）**，这是它的"出厂设置"，定义了：

- **身份**：你是谁？（一个助手、一个专家、一个角色）
- **能力**：你能做什么？（有哪些工具可用）
- **行为规则**：你应该如何回应？（格式、风格、限制）
- **环境信息**：当前时间、可用资源、安全策略等

OpenClaw 的系统提示词是**动态生成**的，包含以下部分：

1. **基础身份**：Agent 的名称、描述、emoji 等
2. **工具描述**：当前可用的所有工具及其参数说明（JSON Schema 格式）
3. **运行时信息**：当前时间、日期、环境变量等
4. **安全提示**：沙箱边界、禁止行为等
5. **格式说明**：如何输出工具调用、如何组织回复

### 2.5 MCP：工具生态的标准协议

MCP 的核心思想：**标准化 AI 与外部世界的交互接口**。

在 MCP 之前，每个 AI 框架都有自己的工具定义方式：
- OpenAI 的 Function Calling
- LangChain 的 Tools
- 各平台自定义的插件系统

MCP 试图统一这一切。一个 MCP 兼容的工具，可以被任何支持 MCP 的 AI 系统使用。

OpenClaw 的**Skills**就是 MCP 的实现：
- 每个 Skill 是一个独立的包，包含工具定义和实现
- Skill 通过 JSON Schema 描述工具的输入输出
- Gateway 负责 Skill 的注册、发现和调用
- Agent 通过标准接口与 Skill 交互，无需关心具体实现

### 3.5 WAL Protocol（写入前置日志协议）

**核心理念**: 在回复之前，先把关键信息写入记忆文件。

**触发条件**:
- ✏️ **修正**: "It's X, not Y" / "Actually..." / "No, I meant..."
- 📍 **专有名词**: 人名、地名、公司名、产品名
- 🎨 **偏好**: 颜色、风格、方法、"我喜欢/不喜欢"
- 📋 **决策**: "Let's do X" / "Go with Y" / "Use Z"
- 🔢 **数值**: 数字、日期、ID、URL

**工作流程**:
```
人类输入 → 扫描关键信息 → 写入 SESSION-STATE.md → 回复人类
```

## 四、典型使用场景

### 4.1 场景一：个人 AI 助手

**使用方式**:
1. 通过 WhatsApp/Telegram/微信发送消息
2. AI 自动识别任务类型
3. 调用相应工具执行（读取文件、执行命令、网络搜索等）
4. 返回结果

**示例**:
```
用户：帮我检查一下今天的日志文件，总结一下我完成了什么

AI 流程：
1. 读取 memory/2026-03-27.md
2. 分析内容
3. 总结完成的任务
4. 发送回复
```

### 4.2 场景二：代码开发助手

**使用方式**:
1. 描述任务需求
2. AI 调用代码执行器（御坂妹妹 11 号）
3. 生成代码并测试
4. 返回结果和解释

**示例**:
```
用户：帮我写一个 Python 脚本，计算斐波那契数列

AI 流程：
1. 识别为代码任务
2. 分派给御坂妹妹 11 号
3. 11 号生成代码
4. 11 号执行测试
5. 汇报结果给御坂大人
```

### 4.3 场景三：信息研究分析

**使用方式**:
1. 提出研究主题
2. AI 调用网络搜索工具
3. 抓取和分析网页内容
4. 生成研究报告

**示例**:
```
用户：搜索 OpenClaw 的官方文档，总结核心功能

AI 流程：
1. 识别为研究任务
2. 分派给御坂妹妹 13 号
3. 13 号调用 web_search 和 web_fetch
4. 分析搜索结果
5. 生成总结报告
```

### 4.4 场景四：博客写作助手

**使用方式**:
1. 提供写作主题
2. AI 调用博客写作技能
3. 生成 Markdown 格式的文章
4. 保存并同步到 Git

**示例**:
```
用户：帮我写一篇关于 OpenClaw 架构的博客文章

AI 流程：
1. 识别为写作任务
2. 调用 hexo-blog 技能
3. 生成文章草稿
4. 保存至 source/_posts/
5. 同步到 Git
```

### 4.5 场景五：系统监控和维护

**使用方式**:
1. 配置定时任务
2. AI 定期检查系统状态
3. 发现问题并告警
4. 自动修复或报告

**示例**:
```cron:
# 每 6 小时检查一次系统状态
openclaw cron add \
  --name "系统监控" \
  --cron "0 */6 * * *" \
  --session main \
  --message "检查系统健康状态"
```

| 维度 | ChatGPT | OpenClaw |
|------|---------|----------|
| **定位** | 聊天机器人 | Agent 运行时平台 |
| **能力** | 生成文本 | 真正执行任务 |
| **记忆** | 会话内临时 | 持久化到磁盘 |
| **工具** | 有限 API | 文件系统、命令、浏览器等 |
| **部署** | 云端 | 本地/私有 |
| **安全** | 受限于平台 | 多层次控制 |

| **可扩展** | 插件生态 | Skills 系统 + ClawHub |

### 5.3 与 AutoGen 对比

| 维度 | AutoGen | OpenClaw |

| **工具** | 需要自行配置 | 预配置 20+ 工具 |
| **持久化** | 需要自行实现 | 内置持久化记忆 |
| **安全性** | 需要自行控制 | 多层次安全模型 |
| **易用性** | 需要编程知识 | 命令 + 图形界面 |

| **多 Agent** | 独立工作空间隔离 |
| **工具丰富** | 20+ 核心工具 + Skills 扩展 |
| **记忆持久** | 三层记忆架构，永不丢失 |
